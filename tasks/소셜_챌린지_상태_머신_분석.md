## 1. 요약

해빛스쿨의 소셜 챌린지는 `social_challenges` 컬렉션을 중심으로 움직이는 별도 상태 머신이다. 개인 챌린지처럼 `users` 내부 맵에 묻혀 있지 않고, 초대 기반 협업/경쟁 도메인을 따로 분리해 둔 구조다.

핵심 흐름:

1. 대시보드에서 단체 목표 또는 1:1 경쟁 챌린지를 생성
2. 서버 `createSocialChallenge`가 입력 검증, 최소 활동 이력 확인, 친구 관계 확인, 스테이크 락업을 처리하고 `pending` 문서를 생성
3. 초대받은 사용자가 `respondSocialChallenge`로 수락/거절
4. 모든 초대 대상이 수락하면 `active`로 전환되고 당일부터 기간이 시작
5. 스케줄 함수 `settleDueSocialChallenges`가 다음 날 00:10 KST에 결산해 `settled` 또는 만료 취소 `cancelled`로 마감

중요한 특징:

- 상태는 `pending`, `active`, `settled`, `cancelled` 네 단계로 비교적 단순하다.
- 단체 목표는 “전원 70% 이상 기록”일 때만 전원 보너스가 지급된다.
- 1:1 경쟁은 양쪽 스테이크를 락업한 뒤 승자독식 구조지만, 한쪽이라도 활동일이 0일이면 무효 처리로 전액 환불된다.
- 어뷰징 방지를 위해 상호 친구 관계, 최근 30일 내 5일 이상 기록, 경쟁 모드 1인 제한, 최대 스테이크 200P 제한이 들어가 있다.

핵심 근거 코드:

- 대시보드 렌더/생성 모달/응답/알림: `js/app.js:7174-7556`
- 생성/응답/결산/스케줄: `functions/index.js:2990-3415`
- 생성 모달/응답 모달 UI: `index.html:1213-1288`
- Firestore 보안 규칙: `firestore.rules:187-194`
- Firestore 인덱스: `firestore.indexes.json:65-78`

## 2. 상태 저장 구조

소셜 챌린지는 문서 하나가 챌린지 하나를 나타낸다.

공통 필드:

| 필드 | 의미 |
|---|---|
| `type` | `group_goal` 또는 `competition` |
| `status` | `pending`, `active`, `settled`, `cancelled` |
| `creatorId` / `creatorName` | 생성자 식별 정보 |
| `invitees` | 아직 응답하지 않았거나 아직 남아 있는 초대 대상 |
| `participants` | 이미 참가 확정된 사용자 목록. 생성자는 생성 시점부터 포함 |
| `durationDays` | 3, 7, 14일 |
| `startDate` / `endDate` | `active` 전환 후 채워지는 KST 기준 날짜 문자열 |
| `expiresAt` | 초대 만료 시각 |
| `createdAt` | 생성 시각 |
| `results` | 결산 결과 맵 |

유형별 추가 필드:

| 유형 | 추가 필드 | 의미 |
|---|---|---|
| `group_goal` | `targetCompletionPct` | 현재는 0.7 고정 |
| `competition` | `stakePoints`, `stakes` | 개인별 락업 스테이크 |

결산 후 `results[uid]`에는 사용자별 `habitPoints`, `activeDays`, `bonusPoints`, `outcome`이 저장되고, 단체 목표일 때는 `completionPct`, `achieved`도 추가된다.

- 생성 문서 구조: `functions/index.js:3090-3108`
- 결산 결과 구조: `functions/index.js:3276-3357`

## 3. 상태 머신

| 상태 | 의미 | 진입 조건 | 다음 전이 |
|---|---|---|---|
| `pending` | 초대 대기 상태 | 생성 직후 | 수락 누적, 전체 수락 시 `active`, 전체 거절/만료 시 `cancelled` |
| `active` | 챌린지 진행 중 | 초대 대상이 모두 수락 | 스케줄 결산 후 `settled` |
| `settled` | 결산 완료 | `endDate`가 지난 뒤 스케줄 결산 성공 | 종료 상태 |
| `cancelled` | 시작 전 취소 | 초대 만료 또는 모든 초대 대상 거절 | 종료 상태 |

핵심 해석:

- 생성자는 `pending` 단계에서도 이미 `participants`에 들어간다.
- 초대를 먼저 수락한 사람도 `participants`로 편입되지만, 남은 초대자가 있으면 상태는 계속 `pending`이다.
- `endDate`는 포함 범위의 마지막 날로 쓰이고, 실제 결산은 그 다음 날 스케줄에서 일어난다.

근거:

- 생성 시 `participants: [uid]`: `functions/index.js:3098-3099`
- 수락 시 `participants` 확장: `functions/index.js:3204-3209`
- 전체 수락 후 `active`: `functions/index.js:3212-3218`
- 만료/거절 취소: `functions/index.js:3157-3183`
- 결산 후 `settled`: `functions/index.js:3360`

## 4-2. 서버 생성 검증

`createSocialChallenge`는 다음을 검증한다.

- 타입은 `group_goal` 또는 `competition`
- 초대 인원은 1~3명
- 기간은 3, 7, 14일만 허용
- 자기 자신 초대 금지
- 경쟁 모드는 정확히 1명만 초대 가능

## 5-2. 만료 처리

응답 시점에 `expiresAt`이 이미 지났다면 서버는:

1. 챌린지를 `cancelled`로 바꾸고
2. 경쟁 모드에서 생성자가 실제로 락업한 스테이크가 있으면 환불한 뒤
3. `deadline-exceeded` 오류를 던진다.

근거:

- 만료 시 취소/환불: `functions/index.js:3157-3165`

## 5-3. 거절 처리

거절하면 해당 사용자는 `invitees`에서 제거된다.

- 아직 다른 초대 대상이 남아 있으면 그대로 `pending`
- 마지막 남은 초대 대상까지 거절하면 `cancelled`
- 경쟁 모드라면 이 시점에 생성자 스테이크를 반환

근거:

- 거절 처리: `functions/index.js:3168-3183`

## 5-4. 수락 처리와 `active` 전환

수락 시 서버는:

1. 수락자도 최근 30일 내 5일 이상 활동했는지 확인
2. 경쟁 모드면 수락자 스테이크를 차감
3. `invitees`에서 제거하고 `participants`에 추가
4. 더 이상 남은 `invitees`가 없으면 `active`로 전환
5. `startDate = todayKST()`
6. `endDate = addDaysKST(startDate, durationDays - 1)`
7. 참가자 모두에게 `challenge_started` 알림 저장

즉 `endDate`는 포함 범위의 마지막 날이다. 3일 챌린지라면 시작일 포함 3일간 유효하다.

근거:

- 수락 검증/스테이크: `functions/index.js:3186-3202`

## 6. 결산 로직

## 6-1. 결산 시점

`settleDueSocialChallenges`는 매일 `00:10 KST`에 실행된다.

스케줄 함수는 먼저 만료된 `pending` 챌린지를 취소하고, 그다음 `status == 'active' && endDate < todayKST()`인 챌린지를 결산한다.

이 조건 때문에 실제 결산은 `endDate` 당일이 아니라 그 다음 날 새벽 이후에 이뤄진다.

근거:

- 스케줄 선언: `functions/index.js:3379-3384`
- 만료 pending 정리: `functions/index.js:3385-3400`
- 결산 대상 조회: `functions/index.js:3402-3411`

## 6-2. 참가자 통계 집계 방식

결산 시 서버는 각 참가자의 `daily_logs`를 `startDate <= date <= endDate` 범위로 읽어:

- `totalPoints`: `awardedPoints.dietPoints + exercisePoints + mindPoints` 합계
- `activeDays`: 해당 기간에 존재하는 `daily_logs` 문서 수

로 계산한다.

중요:

- `activeDays`는 “그 날짜에 기록 문서가 있었는지”만 본다.
- 개인 챌린지처럼 식단/운동/마음 3종을 다 했는지까지 보지 않는다.
- 점수가 0인 기록이어도 문서만 있으면 활동일로 카운트된다.

근거:

- 집계 로직: `functions/index.js:3255-3272`

## 6-3. 단체 목표(`group_goal`)

단체 목표는 각 참가자별로:

- `completionPct = activeDays / durationDays`
- `achieved = completionPct >= 0.7`

를 계산한다.

이후:

- 한 명이라도 기준 미달이면 각자 `achieved` 또는 `missed`로 끝
- 전원이 기준 이상이면 각자 `habitPoints * 20%`를 내림한 만큼 보너스를 받고 `success`로 덮어씀

즉 “내가 70%를 넘겼다”만으로 보너스를 받는 구조가 아니라, 반드시 전원 성공이어야 한다.

근거:

- 달성률 계산: `functions/index.js:3276-3290`
- 전원 성공 보너스: `functions/index.js:3292-3303`

## 6-4. 1:1 경쟁(`competition`)

경쟁 모드는 세 갈래다.

1. 한쪽이라도 `activeDays === 0`
   - 전원 스테이크 전액 환불
   - 둘 다 `void`
2. 양쪽 총점 동점
   - 전원 스테이크 전액 환불
   - 둘 다 `draw`
3. 승패 결정
   - 총점이 높은 쪽이 승자
   - 승자는 `내 스테이크 + 상대 스테이크 + 내 기간 점수 30%`를 받음
   - 결과 맵의 `bonusPoints`에는 “상대 스테이크 + 승리 보너스”만 기록되고, 자기 스테이크 반환분은 암묵적으로 포함되지 않음

근거:

- 무효 처리: `functions/index.js:3310-3321`
- 동점 처리: `functions/index.js:3322-3333`
- 승패 처리: `functions/index.js:3335-3357`

## 7. 어뷰징 방지 장치

현재 코드에 들어 있는 방지 장치는 다음과 같다.

1. 상호 친구 관계 확인  
초대 대상 문서에 생성자 uid가 친구로 등록돼 있어야 챌린지를 만들 수 있다. 일회성 부계정 초대 난이도를 높이는 장치다.

2. 최근 30일 내 최소 5일 활동 이력  
생성자와 수락자 모두 `daily_logs` 기준 최근 30일 내 최소 5일 기록이 있어야 한다.

3. 경쟁 모드 1인 제한  
1:1 경쟁은 상대를 1명만 선택할 수 있고 서버도 다시 확인한다.

4. 스테이크 상한  
경쟁 모드 스테이크는 10P 단위, 최대 200P다.

5. 사전 락업  
생성자와 수락자는 챌린지 시작 전에 각각 스테이크를 차감당한다.

6. 0활동 무효 규칙  
경쟁 모드에서 어느 한쪽이라도 활동일이 0이면 승패로 정산하지 않고 무효 처리로 전액 환불한다.

근거:

- 최소 활동/친구/스테이크 검증: `functions/index.js:3029-3070`, `functions/index.js:3186-3202`
- 0활동 무효 처리: `functions/index.js:3310-3321`

근거:

## 8-3. 알림 처리

서버는 세 종류의 알림을 저장한다.

- `challenge_invite`
- `challenge_started`
- `challenge_settled`

하지만 대시보드 진입 시 토스트로 확인하는 함수는 `challenge_invite`, `challenge_settled`만 읽는다. 즉 서버는 `challenge_started`를 기록하지만, 현재 클라이언트의 배경 알림 확인 흐름에서는 이 타입을 소비하지 않는다.

근거:

- 시작 알림 저장: `functions/index.js:3220-3231`
- 배경 알림 조회 대상: `js/app.js:7524-7529`

## 9. Firestore 규칙과 인덱스

`social_challenges` 규칙은 다음처럼 돼 있다.

- 읽기: 로그인했고 `participants` 또는 `invitees`에 내 uid가 있으면 허용
- 생성: 로그인했고 `creatorId == auth.uid`이며 `status == 'pending'`이면 허용
- 수정/삭제: 전부 금지

인덱스는 최소 두 개가 준비돼 있다.

- `status + endDate`: 스케줄 결산 조회용
- `creatorId + status`: 생성자 중복 챌린지 확인용

근거:

- 규칙: `firestore.rules:187-194`
- 인덱스: `firestore.indexes.json:65-78`

## 10. 구현상 주의점과 관찰

## 10-1. 중복 방지는 생성자 기준만 본다

현재 서버는 “내가 만든 진행 중 챌린지”만 막는다. 이미 다른 사람이 만든 챌린지에 참가 중인 사용자가 또 새 챌린지를 만들거나, 여러 챌린지에 겹쳐 초대받는 구조는 별도로 차단하지 않는다.

근거:

- `functions/index.js:3073-3081`

## 10-2. `activeDays`는 느슨한 활동 판정이다

단체 목표 성공률과 경쟁 무효 판정 둘 다 `daily_logs` 문서 존재 여부만으로 계산된다. 즉 “완전한 습관 달성일”이 아니라 “해당 날짜에 무언가 기록한 날” 기준에 더 가깝다.

근거:

- `functions/index.js:3255-3272`

## 10-3. 결산/취소 이력은 메인 카드에서 사라진다

`settled`와 `cancelled`는 대시보드 카드에 남지 않기 때문에, 사용자는 주로 토스트와 내부 결과 맵에 의존해 결과를 확인하게 된다. 히스토리 관점에서는 별도 조회 화면이 없는 셈이다.

근거:

- `js/app.js:7186-7200`

## 10-4. Firestore 규칙 계층에서 서버 검증이 완전히 강제되지는 않는다

현재 규칙은 signed-in 사용자가 `creatorId == 자기 자신`, `status == 'pending'`만 맞추면 `social_challenges` 문서를 직접 생성할 수 있게 열려 있다. 앱 정상 경로는 callable function을 쓰지만, 규칙만 보면 최소 활동 검증, 친구 관계 검증, 스테이크 락업 같은 서버 비즈니스 규칙이 Firestore create 단계에서 강제되지는 않는다.

즉 현 구조는 “앱은 서버 함수를 사용한다”는 전제를 두고 설계돼 있고, 규칙 계층만으로는 동일한 무결성을 보장하지 않는다.

근거:

- Firestore create 허용 범위: `firestore.rules:191-193`
- 서버 검증 본체: `functions/index.js:3029-3088`

## 11. 정리

소셜 챌린지는 해빛스쿨 안에서 가장 명확하게 상태 전이가 드러나는 별도 도메인 모델이다. 생성-초대-수락-활성-결산의 흐름이 잘 분리돼 있고, 경쟁 모드에는 최소한의 어뷰징 방지 장치도 들어 있다.

다만 실제 운영 관점에서는 다음 세 가지가 특히 중요하다.

1. 중복 참여 제한이 생성자 기준에만 걸려 있다는 점
2. 활동 판정이 문서 존재 기준이라 상대적으로 느슨하다는 점
3. Firestore 규칙 계층에서 서버 비즈니스 검증이 완전히 봉인돼 있지 않다는 점

다음 분석 단계에서는 이 소셜 도메인과 맞닿아 있는 `gallery`, `notifications`, `admin` 영역 중 하나를 이어서 보면 흐름이 자연스럽다.
