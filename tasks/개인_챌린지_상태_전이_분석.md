# 해빛스쿨 개인 챌린지 상태 전이 분석

작성일: 2026-04-03  
분석 방식: 코드 정적 분석  
분석 범위: 개인 챌린지 시작, 진행도 누적, 만료 처리, 보상 수령, 실패 정산, 포기, 레거시 호환, 사용자 표시 상태

---

## 1. 요약

해빛스쿨의 개인 챌린지는 `users/{uid}` 문서 안의 `activeChallenges` 맵으로 관리되는 클라이언트+Cloud Functions 혼합 상태 머신이다.

핵심 흐름:

1. 자산 탭에서 챌린지 시작
2. 저장 시 `updateChallengeProgress()`가 진행도 누적
3. 종료 시점에 80% 이상이면 `claimable`, 미만이면 실패 정산 또는 제거
4. 사용자가 직접 보상 수령 또는 포기

중요한 특징:

- 티어는 `mini`, `weekly`, `master`로 분리되고, 티어별 1개씩만 동시 진행 가능하다.
- `expired`는 내부 전용 중간 상태에 가깝고, UI에서는 거의 보이지 않는다.
- 실패 정산은 클라이언트와 서버가 나눠 처리해 로그가 이중으로 남을 가능성이 있다.

핵심 근거 코드:

- 시작/진행/만료/수령/포기: `js/blockchain-manager.js:458-933`
- 서버 시작/수령/실패 정산: `functions/index.js:1255-1629`
- 자산 탭 UI 상태 렌더: `js/app.js:1795-1906`
- 로그인 후 만료 정산 진입: `js/auth.js:304-311`

## 4. 상태 테이블

| 상태 | 저장 위치 | 사용자에게 보임 | 다음 전이 |
|---|---|---|---|
| 없음 | 해당 티어 키 없음 | 시작 카드만 보임 | `startChallenge` |
| `ongoing` | `activeChallenges[tier]` | 진행 카드로 보임 | 진행도 증가, `claimable`, `expired`, 삭제, 포기 |
| `claimable` | `activeChallenges[tier].status` | 성공 카드로 보임 | `claimChallengeReward` 후 삭제 |
| `expired` | `activeChallenges[tier].status` | UI에서 거의 숨김 | `settleExpiredChallenges`로 실패 정산 |
| 삭제됨 | 티어 키 삭제 | 비활성 카드 해제 | 새 챌린지 시작 가능 |

핵심 해석:

- 사용자 관점에서 보이는 상태는 사실상 `ongoing`과 `claimable` 두 가지다.
- `expired`는 정산 대기용 내부 상태에 가깝다.

근거:

- UI 필터: `js/app.js:1810-1812`
- 만료 처리: `js/blockchain-manager.js:743-761`

## 5. 시작 흐름

## 5-1. 클라이언트 검증

`startChallenge30D()`는 다음을 확인한다.

- 로그인 여부
- 챌린지 ID 매핑
- 티어별 최소/최대 HBT 예치량
- 소수점 2자리 제한
- 예치형일 경우 온체인 HBT 잔액
- 가스 부족 시 `prefundWallet`로 자동 충전 시도

근거:

- `js/blockchain-manager.js:458-554`

## 5-2. 온체인 예치

예치형 챌린지는 시작 전에 `stakeForChallenge(rawAmount)` 트랜잭션을 먼저 보낸다. 성공하면 `stakeTxHash`를 Cloud Function으로 넘긴다.

근거:

- `js/blockchain-manager.js:505-543`

## 5-3. 서버 시작 처리

서버 `startChallenge`는:

1. 로그인 확인
2. 챌린지 ID 매핑
3. 최소/최대 예치량 재검증
4. 예치형이면 `stakeTxHash` 영수증 검증
5. 같은 티어에 `ongoing` 또는 `claimable`이 있으면 거부
6. `challengeData` 생성 후 `activeChallenges.{tier}`에 저장

근거:

- `functions/index.js:1255-1385`

## 5-4. 시작 당일 1일 선반영

서버는 시작 시점의 오늘 `daily_logs/{uid}_{startDate}`를 읽고, 이미 `diet && exercise && mind`를 만족하면:

- `completedDays = 1`
- `completedDates = [startDate]`

로 저장한다.

근거:

- `functions/index.js:1329-1337`

## 6. 진행도 누적 흐름

진행도 누적은 `updateChallengeProgress()`가 담당하고, 일일 기록 저장 직후 백그라운드로 호출된다.

근거:

근거:

## 6-2. 중복 방지

같은 날짜는 `completedDates.includes(today)`로 중복 카운트를 막는다.

근거:

## 6-3. 진행 중 카운트

조건을 만족하면:

- `completedDates.push(today)`

로 업데이트된다.

근거:

- `js/blockchain-manager.js:693-696`

## 7. 종료 시점 판정

## 7-1. 내부 종료 기준

`updateChallengeProgress()`와 `settleExpiredChallenges()` 모두 `today >= challenge.endDate`를 “기한 종료”로 본다.

근거:

- `js/blockchain-manager.js:637`
- `js/blockchain-manager.js:749`

## 7-2. 중요한 해석: `endDate`는 사실상 경계일이다

서버 시작 로직은 `endDate = startDate + duration`으로 계산한다.

예:

- 3일 챌린지 시작일이 `2026-04-03`
- 서버 저장 `endDate`는 `2026-04-06`
- 하지만 `today >= endDate`이면 더 이상 누적하지 않으므로 실제 인정일은 `04-03`, `04-04`, `04-05`의 3일이다.

근거:

- 서버 계산: `functions/index.js:1323-1327`
- 종료 판정: `js/blockchain-manager.js:637`
- UI 표시: `js/app.js:1871`

즉 `endDate`는 UI에 표시되는 “마지막 인증 가능일”이라기보다 “그 다음 경계일”처럼 동작한다.

근거:

- `js/auth.js:304-311`

## 9. 실패 분기: `expired` 또는 즉시 삭제

## 9-1. 무예치 챌린지 실패

예치가 없는 실패(`mini`)는 종료 시 즉시 제거된다.

- `activeChallenges.{tier} = null`
- 실패 로그를 `challenge_settlement` 타입으로 남김

근거:

- `js/blockchain-manager.js:651-668`

## 9-2. 예치형 챌린지 실패

예치형 실패(`weekly`, `master`)는 종료 시 즉시 소각하지 않고 우선 `expired`로 마킹한다.

- `status = 'expired'`
- 실패 로그를 `challenge_settlement` 타입으로 남김

근거:

- `js/blockchain-manager.js:648-668`

이후 `settleExpiredChallenges()`가 실제 실패 정산을 수행한다.

## 10. 만료 정산 흐름

`settleExpiredChallenges()`는 아래 시점에 호출된다.

- 로그인 후 블록체인 모듈 로드 완료 직후
- 자산 탭 진입 후 블록체인 모듈 로드 완료 직후

근거:

- 로그인 후: `js/auth.js:304-311`
- 자산 탭 후: `js/app.js:2065-2069`

## 10-1. 서버 캐시 회피

이 함수는 `getDocFromServer(userRef)`를 사용해 오래된 캐시 때문에 `completedDays`가 낮게 보이는 문제를 피하려고 한다.

근거:

- `js/blockchain-manager.js:733-736`

## 10-2. 정산 분기

대상:

- `status === 'ongoing' || status === 'expired'`
- 그리고 `today >= endDate` 또는 이미 `expired`

근거:

- `js/blockchain-manager.js:742-749`

분기:

- `successRate >= 0.8`
  - `claimable`로 전환
- `successRate < 0.8`
  - 예치형이면 `settleChallengeFailure` Cloud Function 호출
  - 무예치면 그냥 삭제

근거:

- `js/blockchain-manager.js:757-784`

## 10-3. 실패 정산 서버 처리

`settleChallengeFailure`는:

- `resolveChallenge(user, false)` 호출
- 온체인 기준 50% 소각, 50% 반환
- `activeChallenges.{tier}` 삭제
- `blockchain_transactions`에 `challenge_failure` 기록

근거:

- `functions/index.js:1540-1629`

## 11. 보상 수령 흐름

`claimChallengeReward(tier)`는 클라이언트가 Cloud Function `claimChallengeReward`를 호출하는 얇은 래퍼다.

근거:

- `js/blockchain-manager.js:817-861`

## 11-1. 서버 수령 조건

서버는 해당 티어 챌린지가 `claimable`일 때만 수령을 허용한다.

근거:

- `functions/index.js:1413-1417`

## 11-2. 보상 매트릭스

예치형:

- 100%: 원금 반환 + 기본 포인트 + HBT 보너스
- 80% 이상 100% 미만: 원금 반환 + 기본 포인트

무예치형:

- 100%: 기본 포인트 전액
- 80% 이상 100% 미만: 기본 포인트 50%

근거:

- `functions/index.js:1432-1445`

보너스율:

- `weekly`: 예치금의 50%
- `master`: 예치금의 200%

근거:

- `functions/index.js:1434-1435`
- `functions/index.js:1468-1470`

## 11-3. 온체인 성공 정산

예치형이고 80% 이상이면 서버는:

1. `resolveChallenge(user, true)`로 원금 반환
2. 100%인 경우 추가 HBT 보너스를 `mint()`로 민팅
3. `activeChallenges.{tier}` 삭제
4. `challenge_settlement` 성공 로그 기록

근거:

- `functions/index.js:1450-1522`

## 11-4. `NoStakeFound` 예외 처리

- `rewardHbt = 0`
- Firestore 정리와 포인트 지급은 계속 진행

근거:

## 13. 사용자에게 보이는 상태 vs 내부 상태

## 13-1. UI는 `expired`를 거의 보여주지 않는다

자산 탭 UI는 `ongoing`과 `claimable`만 활성 챌린지로 간주한다.

근거:

- `js/app.js:1810-1812`

따라서 `expired`는 내부적으로는 존재해도 카드에 보이지 않는다.

## 13-2. 거래 기록은 실패 케이스를 완전히 이해하지 못한다

자산 탭 거래 목록은 `conversion`, `staking`, `challenge_settlement`, `withdrawal`만 전용 라벨/아이콘을 갖는다.

근거:

- `js/app.js:1947-1985`

그런데 서버 실패 정산 로그는 `challenge_failure` 타입이고, `amount` 필드 없이 `burned`, `returned`만 저장한다.

근거:

- `functions/index.js:1609-1614`

따라서 현재 거래 목록에서는 `challenge_failure`가:

- 전용 라벨 없이 raw type처럼 보일 수 있고
- 금액도 `0 HBT`처럼 보일 가능성이 높다

라는 해석이 가능하다.

## 13-3. 오늘 HBT 델타도 성공 정산만 합산한다

자산 탭의 오늘 HBT 델타는 `conversion` 성공과 `challenge_settlement` 성공만 합산한다.

근거:

- `js/app.js:1514-1525`
- `js/app.js:1640-1646`

즉 실패 정산으로 반환된 HBT는 온체인 잔액에는 반영되더라도 “오늘 +HBT” 지표에는 직접 반영되지 않는다.

## 14. 분석 중 눈에 띄는 위험 포인트

## 14-1. `expired` 상태가 시작 차단 조건에서 빠져 있다

서버 `startChallenge`는 같은 티어의 `ongoing` 또는 `claimable`만 차단한다.

근거:

- `functions/index.js:1313-1316`

UI도 `expired`를 활성 챌린지로 보지 않아서 해당 티어 카드를 다시 열어둘 수 있다.

근거:

- `js/app.js:1810-1812`
- `js/app.js:1894-1899`

정적 분석 기준 추론:

- 정산 전 `expired` 챌린지가 남아 있는 짧은 구간에
- 같은 티어 새 챌린지가 시작되면
- 서버가 `activeChallenges.{tier}`를 새 데이터로 덮어쓸 가능성이 있다

실사용에서는 로그인/자산 탭 진입 시 `settleExpiredChallenges()`가 먼저 돌도록 의도돼 있지만, 코드상 차단 조건 자체는 `expired`를 포함하지 않는다.

## 14-2. 실패 로그가 중복으로 남을 가능성

실패 시:

- `updateChallengeProgress()`가 `challenge_settlement` 실패 로그를 남길 수 있고
- 이후 `settleExpiredChallenges()`도 다시 `challenge_settlement` 실패 로그를 남긴다
- 서버는 별도로 `challenge_failure` 성공 로그도 남긴다

근거:

- 첫 실패 로그: `js/blockchain-manager.js:660-668`
- 만료 정산 로그: `js/blockchain-manager.js:791-797`
- 서버 실패 로그: `functions/index.js:1609-1620`

즉 정산 경로에 따라 실패 로그가 2~3개로 보일 수 있다.

## 14-3. 표시 날짜와 실제 판정 경계가 어긋난다

UI는 `startDate ~ endDate`를 그대로 보여주지만, 실제 누적은 `today < endDate`인 날까지만 가능하다.

근거:

- UI 표시: `js/app.js:1871`
- 내부 판정: `js/blockchain-manager.js:637`
- 서버 계산: `functions/index.js:1323-1327`

따라서 사용자는 끝나는 날짜를 하루 더 길게 인식할 수 있다.

## 15. 결론

개인 챌린지는 겉으로 보기엔 단순한 “시작 → 완료 → 보상” 흐름이지만, 실제 코드는 아래 층으로 나뉜다.

1. 시작 전 온체인 예치 검증
2. 저장 시점의 일일 진행도 누적
3. 종료 후 `claimable`/`expired`/삭제 분기
4. 수령 시 온체인 원금 반환과 보너스 민팅
5. 포기/실패 시 50% 소각 정산

핵심적으로 기억할 점은 세 가지다.

1. 내부 상태는 `ongoing`, `claimable`, `expired`, 삭제까지 존재하지만 UI는 주로 `ongoing`과 `claimable`만 보여준다.
2. `endDate`는 표시상 종료일처럼 보이지만 실제로는 “더 이상 누적하지 않는 경계일”에 가깝다.
3. 실패 정산과 거래 로그는 사용자 표시보다 내부 로직이 더 복잡해서, 운영/디버깅 때는 `activeChallenges`와 `blockchain_transactions`를 함께 봐야 한다.
