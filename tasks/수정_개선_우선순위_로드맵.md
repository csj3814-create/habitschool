# 수정·개선 우선순위 로드맵

> 기준: 2026-04-03까지 작성한 분석 문서 종합
>
> 상태: 완료

## 1. 목적

이 문서는 지금까지의 정적 분석 결과를 바탕으로, 해빛스쿨을 어떤 순서로 고쳐야 리스크를 가장 빨리 낮추고 이후 작업 난이도도 줄일 수 있는지 정리한 실행 우선순위 제안서다.

핵심 원칙은 아래 네 가지다.

1. 사용자 프라이버시와 권한 문제를 먼저 막는다.
2. 포인트/HBT/관리자 집계처럼 신뢰를 해칠 수 있는 경제 정합성을 먼저 바로잡는다.
3. 그 다음 상태 머신과 알림처럼 누적 복잡도를 키우는 구조를 정리한다.
4. 마지막에 UX 정리와 운영 편의 기능을 다듬는다.

## 2. 우선순위 판단 기준

이번 우선순위는 아래 기준으로 정했다.

- 사용자 피해 가능성: 개인정보, 공개 범위, 악용 가능성
- 자산 신뢰성: 포인트/HBT/보상 이력이 실제와 다르게 보일 위험
- 운영 리스크: 관리자 화면이 잘못된 판단을 유도하는 정도
- 수정 파급도: 먼저 고치면 뒤 작업이 쉬워지는 구조적 중요도
- 회귀 위험: 늦게 손댈수록 데이터/상태가 더 꼬이는 항목인지 여부

## 3. 전체 추천 순서

1. P0: 보안·공개 범위·권한 구멍 차단
2. P0: 경제 원장 최소 정합성 회복
3. P1: 챌린지/미션 상태 머신 정리
4. P1: 알림 모델 단일화
5. P2: 관리자/운영 관제 품질 개선
6. P2: UX 표현 정합성 정리

## 4. P0 — 지금 가장 먼저 막아야 할 것

### P0-1. `daily_logs` 전체 공개 구조 분리

왜 먼저인가:

- 현재 갤러리는 `daily_logs`를 그대로 공개 피드 원본으로 쓰고 있다.
- Firestore 규칙상 `daily_logs`는 `allow read: if true`라서 실제 공개 범위가 UI 인식보다 훨씬 넓다.
- 건강 기록, 식단, 혈당, 혈압, 수면/마음 데이터가 같은 문서에 섞여 있기 때문에 프라이버시 리스크가 가장 크다.

권장 방향:

- 갤러리용 공개 문서를 `daily_logs`와 분리
- 공개 필드 화이트리스트 정의
- 기존 `daily_logs`는 본인/관리자 전용으로 축소
- 공개 여부 토글이 필요하면 새 공개 컬렉션 기준으로 설계

선행 이유:

- 이 문제를 늦게 고치면 이후 알림, 커뮤니티, 관리자, 추천 기능이 모두 공개 데이터 설계를 잘못된 전제 위에 쌓게 된다.

### P0-2. Firestore 규칙 과개방 차단

왜 먼저인가:

- `notifications`는 로그인 사용자면 임의 알림 문서를 만들 수 있다.
- `social_challenges`도 규칙 레벨에서 callable 검증을 완전히 강제하지 못한다.
- `daily_logs`의 반응/댓글 업데이트 규칙도 의도보다 넓게 열려 있다.

권장 방향:

- `notifications` 생성은 Cloud Functions 전용으로 축소
- `social_challenges` 생성/상태 전이는 서버 쓰기 중심으로 재정의
- `daily_logs`의 반응/댓글 변경 가능 필드를 더 엄격히 분리
- 필요하면 공개용 컬렉션과 원본 로그 컬렉션을 분리해 규칙을 단순화

### P0-3. 경제 정합성의 최소선 복구

왜 먼저인가:

- 포인트와 HBT는 이 앱의 핵심 신뢰 자산이다.
- 지금은 관리자 화면과 사용자 자산 탭이 같은 값을 다른 의미로 보여 준다.
- 통합 포인트 지급 이력도 실제 지급을 완전하게 복원하지 못한다.

이번 단계에서 최소한 바로잡아야 할 것:

- `challenge_settlement` 문서에 `rewardPoints` 저장
- 관리자 수동 조정, 반응 포인트, 가입 축하, 추천 보너스의 기록 전략 정리
- 관리자 HBT 열을 “현재 보유”가 아닌 “누적 획득” 또는 “레거시 값”으로 명확히 분리
- `meta/communityStats.totalHbtMinted`의 실제 계산 경로 명확화

주의:

- 여기서 바로 완전한 이중원장 재설계까지 가지 말고, 우선 “같은 데이터를 다르게 읽는 문제”와 “기록이 빠지는 문제”부터 막는 것이 좋다.

## 5. P1 — 신뢰를 해치는 구조적 불일치 정리

### P1-1. 경제 모델 단일화

목표:

- 아래 네 개념을 명시적으로 분리하는 것
  - 현재 포인트 잔액
  - 현재 온체인 HBT 잔액
  - 누적 획득 HBT
  - 포인트 지급 이벤트 원장

권장 방향:

- `users`는 현재 상태 스냅샷
- `blockchain_transactions`는 HBT 이벤트 로그
- 포인트 지급은 별도 원장 컬렉션 또는 공통 이벤트 스키마 도입
- 관리자 화면은 파생 집계가 아니라 명시적인 소스 라벨을 표시

왜 P1인가:

- 보안 구멍을 막은 뒤 바로 이 작업을 해야 이후 챌린지/알림/운영 기능 수정이 같은 자산 모델 위에서 정렬된다.

### P1-2. 개인 챌린지 상태 머신 정리

핵심 이슈:

- `expired`가 내부 상태인데 UI/시작 차단 조건에서 빠져 있다.
- 실패 정산 로그가 `challenge_settlement`와 `challenge_failure`로 나뉘어 해석이 어긋난다.
- 클라이언트와 서버가 실패/수령 처리를 나눠 들고 있어 중복 또는 레이스 조건 여지가 있다.

권장 방향:

- 상태 전이 소유권을 서버 중심으로 단순화
- 성공/실패/포기 로그 타입을 명확히 정리
- 사용자 UI 상태명과 내부 상태명을 맞춤

### P1-3. 소셜 챌린지 상태 머신 정리

핵심 이슈:

- 생성 검증은 서버에 있지만 규칙 레벨이 충분히 닫혀 있지 않다.
- `activeDays`가 “완전 인증일”이 아니라 문서 존재일 수라 의미가 느슨하다.
- 중복 제한과 무효 처리 기준이 일부 한쪽 방향만 본다.

권장 방향:

- 참가 조건, 활동일 계산, 중복 방지, 결산 결과를 모두 명시적 함수로 고정
- 관리자/사용자 설명 문구와 실제 판정 로직을 일치시킴

### P1-4. 주간 미션 계산식 정리

핵심 이슈:

- 진행 중 달성률과 아카이브 후 `completionRate` 계산식이 다르다.
- 렌더 시점 아카이브라서 첫 방문 타이밍에 따라 체감 상태가 흔들릴 수 있다.

권장 방향:

- 달성률 정의를 하나로 통일
- 주차 전환을 더 명시적인 배치/서버 이벤트로 옮길지 검토

## 6. P1 — 알림 모델 재정의

### P1-5. `notifications`와 FCM의 역할 통합

왜 필요한가:

- 지금은 인앱 토스트와 푸시가 서로 다른 이벤트 흐름을 쓴다.
- 어떤 알림은 문서만 있고, 어떤 알림은 푸시만 있고, 읽음 상태도 기기별 로컬스토리지에만 있다.

권장 방향:

- 알림 이벤트 원본을 하나로 정의
- 그 원본에서 인앱 inbox와 FCM 푸시를 파생
- `read`를 Firestore 기준으로 복원할지 결정
- `challenge_started`처럼 생산자/소비자 타입이 어긋난 항목 정리

우선순위가 P1인 이유:

- 보안과 경제 정합성 다음 단계에서 사용자 신뢰와 UX 일관성을 크게 올릴 수 있다.

## 7. P2 — 운영 화면과 UX 표현 정리

### P2-1. 관리자 화면의 부분 집계 제거

핵심 이슈:

- 공용 캐시가 최근 500건 `daily_logs`만 읽어 활동 지표와 회원 상세 30일 기록이 부분 집계일 수 있다.

권장 방향:

- 탭별 쿼리 분리
- 활동 통계는 집계 문서나 서버 계산으로 이동
- 회원 모달은 사용자별 직접 조회로 변경

### P2-2. 관리자 자산 용어 정리

권장 방향:

- `포인트`, `온체인 HBT`, `누적 획득 HBT`, `보상 지급 이력`을 UI 라벨에서 분리
- “총 발행”, “누적 획득”, “현재 보유”를 같은 HBT 표기 아래 섞지 않기

### P2-3. 갤러리/커뮤니티 UX 정리

권장 방향:

- 공개 여부, 신고 상태, 차단 적용 범위를 사용자에게 더 명확히 보이기
- 현재의 “차단은 개인 숨김” 구조를 UI 문구에 맞게 설명하거나 서버 제재 모델로 확장

### P2-4. 알림함 여부 결정

권장 방향:

- 계속 토스트만 갈지
- 아니면 `notifications`를 실제 inbox로 승격할지

이 결정은 알림 모델 통합 이후에 하는 편이 낫다.

## 8. 추천 작업 묶음

실제 구현은 아래 묶음으로 진행하는 편이 안전하다.

### 1차 묶음: 보안·권한 긴급 패치

- `daily_logs` 공개 범위 축소 또는 공개 컬렉션 분리
- `notifications` 규칙 축소
- `social_challenges` 규칙 축소
- 반응/댓글 규칙 정리

### 2차 묶음: 경제 정합성 패치

- `challenge_settlement.rewardPoints` 저장
- 관리자 수동 조정/추천/가입 축하/리액션의 기록 정책 통일
- 관리자 HBT 라벨과 데이터 소스 분리
- `totalHbtMinted` 계산 경로 고정

### 3차 묶음: 상태 머신 정리

- 개인 챌린지
- 소셜 챌린지
- 주간 미션

### 4차 묶음: 알림 시스템 정리

- 인앱/푸시 단일 이벤트 소스 정의
- `read` 모델 복구 여부 결정
- 알림 타입 소비 불일치 제거

### 5차 묶음: 운영 UI 정리

- 관리자 부분 집계 제거
- 자산/거래 이력 용어 정리
- 갤러리/알림 UX 정리

## 9. 지금 바로 시작하면 좋은 첫 작업

하나만 고른다면 아래 순서를 추천한다.

1. `daily_logs` 공개 범위와 공개 데이터 모델 분리
2. `notifications` / `social_challenges` 보안 규칙 축소
3. `challenge_settlement.rewardPoints` 저장과 관리자 경제 탭 정합성 복구

이 세 가지를 먼저 하면 프라이버시, 악용 가능성, 자산 신뢰성의 가장 큰 구멍을 한 번에 줄일 수 있다.

## 10. 결론

해빛스쿨은 기능이 많은 대신 핵심 데이터가 몇 군데에서 “빠르게 붙은 흔적”이 남아 있다.

지금 가장 중요한 것은 기능 추가보다 아래 두 축을 먼저 바로잡는 것이다.

- 누가 무엇을 읽고 쓸 수 있는가
- 포인트/HBT가 어디서 확정되고 어디에 기록되는가

이 두 축이 정리되면 나머지 챌린지, 알림, 관리자 UX 개선은 훨씬 안전하고 예측 가능하게 진행할 수 있다.
