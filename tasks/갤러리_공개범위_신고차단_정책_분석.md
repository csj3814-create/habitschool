# 해빛스쿨 갤러리 공개 범위 및 신고·차단 정책 분석

작성일: 2026-04-03  
분석 방식: 코드 정적 분석  
분석 범위: 갤러리 진입, 공개 피드 로딩 범위, `daily_logs` 공개 방식, 반응/댓글 정책, 신고/차단 처리, 관리자 후속 조치, Firestore 규칙

---

## 1. 요약

해빛스쿨의 갤러리는 별도 게시물 컬렉션이 아니라 `daily_logs` 자체를 공개 피드 원본으로 사용하는 구조다. 즉 “기록 저장”과 “갤러리 게시”가 사실상 같은 저장소를 공유한다.

핵심 흐름:

1. 사용자가 하루 기록을 `daily_logs/{uid}_{date}`에 저장
2. 갤러리 탭은 최근 30일의 `daily_logs`를 읽어 카드로 렌더
3. 로그인 사용자는 반응/댓글/신고/차단 가능
4. 게스트는 읽기 전용으로 둘러보기 가능
5. 신고는 `reports` 컬렉션에 쌓이고, 관리자가 수동으로 처리하거나 게시물을 삭제

중요한 특징:

- 갤러리는 비로그인 사용자도 의도적으로 접근 가능하다.
- 공개 여부를 고르는 `isPublic`, `shareToGallery`, 비공개 토글 같은 필드는 현재 없다.
- UI는 일부 필드만 보여 주지만, Firestore 규칙상 `daily_logs` 전체 문서는 누구나 읽을 수 있다.
- 차단은 “내 피드에서 해당 사용자 게시물을 숨기는 클라이언트 필터”에 가깝고, 플랫폼 차원의 상호 차단은 아니다.
- 신고는 자동 숨김이나 자동 제재로 이어지지 않고 관리자 수동 처리 중심이다.

핵심 근거 코드:

- 게스트 갤러리 진입: `index.html:161-169`
- 갤러리 탭 로그인 예외: `js/app.js:2029-2031`
- 저장 시 `daily_logs` 생성: `js/app.js:4373-4389`
- 갤러리 로딩: `js/app.js:5691-5877`
- 신고/차단: `js/app.js:5240-5303`
- Firestore 규칙: `firestore.rules:73-92`, `firestore.rules:160-165`
- 관리자 신고 처리: `admin.html:1047-1092`

## 2. 공개 범위의 기본 구조

## 2-1. 갤러리는 비로그인 접근을 전제로 설계돼 있다

게스트 환영 모달에는 “갤러리 둘러보기” 버튼이 따로 있고, 실제 탭 전환 함수도 로그인 없이 `gallery` 탭만 열 수 있게 예외 처리돼 있다.

근거:

- 게스트 진입 버튼: `index.html:169`
- 로그인 없이 허용되는 유일한 탭: `js/app.js:2029-2031`

즉 제품 정책상 갤러리는 처음부터 공개 피드로 의도된 기능이다.

## 2-2. 공개 피드 원본은 `daily_logs`다

하루 기록 저장 시 앱은 별도 게시물 컬렉션이 아니라 `daily_logs` 문서에 바로 아래 정보를 쓴다.

- `userId`, `userName`, `date`, `timestamp`
- `awardedPoints`
- `metrics`(체중, 혈당, 혈압)
- `diet` 사진/썸네일 URL
- `dietAnalysis`
- `exercise` 미디어 목록
- `steps` 스크린샷/걸음 데이터
- `sleepAndMind`(수면 캡처, 분석, 명상 체크, 감사일기)

근거:

- 저장 페이로드: `js/app.js:4373-4384`

이 문서가 그대로 갤러리, 반응, 댓글, 신고 대상의 기준 문서가 된다.

## 3. 무엇이 갤러리에 올라오는가

## 3-1. 공개 전용 토글은 없다

현재 코드에는 게시물 공개 여부를 분리하는 필드가 없다.

- 저장 시 `saveData`에 공개 여부 플래그가 없음
- Firestore 규칙에도 `public/private` 조건이 없음
- UI에도 “갤러리에 공개” 체크박스가 없음

근거:

- 저장 데이터: `js/app.js:4373-4384`
- `daily_logs` 규칙: `firestore.rules:73-92`

즉 현재 구조에서는 “기록을 저장하면 갤러리 원본 문서도 함께 생긴다”고 보는 편이 정확하다.

## 3-2. 실제 피드는 최근 30일만 읽는다

갤러리 로더는 로그인 사용자와 게스트 모두 최근 30일의 `daily_logs`만 읽는다.

- 로그인 사용자: Firestore SDK로 최근 30일 쿼리
- 게스트: Firestore REST `runQuery`로 최근 30일 쿼리

근거:

- 로그인 쿼리: `js/app.js:5815-5825`
- 게스트 REST 쿼리: `js/app.js:5707-5738`, `js/app.js:5790-5799`

다만 이것은 “현재 UI가 불러오는 범위”일 뿐이고, 규칙 레벨에서 읽기 범위가 30일로 제한되지는 않는다.

## 3-3. 카드로 보이기 위한 조건

클라이언트는 `hasMediaForFilter()`로 표시 후보를 거른다.

주요 기준:

- 식단 사진 URL
- 운동 사진/영상 또는 목록
- 수면 캡처
- 감사일기 텍스트
- 명상 체크
- 걸음수(`steps.count > 0`)

근거:

- 필터 함수: `js/app.js:5518-5549`

하지만 카드 렌더 함수 `buildGalleryCard()`는 실제로 식단/운동/수면 이미지와 감사일기 텍스트 중심으로 그린다. 걸음수만 있는 기록은 필터 단계에서는 후보가 될 수 있지만, 최종 카드 생성에서는 표시 콘텐츠가 없어 `null`이 될 수 있다.

근거:

- 카드 렌더: `js/app.js:6220-6243`
- 표시 미디어 수집: `js/app.js:6121-6216`

## 4. UI에 보이는 범위와 실제 읽기 범위의 차이

이 부분이 가장 중요하다.

## 4-1. UI는 일부만 보여 준다

갤러리 카드는 주로 아래만 노출한다.

- 식단 사진
- 운동 사진/영상
- 수면 기록 캡처
- 감사일기 텍스트
- 닉네임, 날짜, 스트릭
- 반응/댓글

근거:

- 미디어 렌더: `js/app.js:6129-6213`
- 카드 조립: `js/app.js:6220-6340`

## 4-2. 하지만 Firestore 규칙은 `daily_logs` 전체 읽기를 열어 둔다

`daily_logs` 문서 규칙은 다음처럼 돼 있다.

- `allow read: if true`

즉 로그인 여부와 무관하게 전체 문서 읽기가 허용된다.

근거:

- `firestore.rules:74-76`

이 때문에 UI에 안 보이는 아래 필드도 문서 읽기 자체로는 접근 가능하다.

- `metrics`(체중, 혈당, 혈압)
- `dietAnalysis`
- `steps.imageHash`, `steps.screenshotUrl`
- `sleepAndMind.sleepAnalysis`
- 기타 저장된 부가 필드

또 게스트 REST 조회도 field mask 없이 전체 문서를 가져온다.

근거:

- REST 조회 본문: `js/app.js:5710-5722`
- Firestore 응답 전체 변환: `js/app.js:5741-5759`

## 4-3. AI 분석 값은 카드 DOM에도 실린다

식단 분석, 운동 분석, 수면 분석은 카드 내부 `data-ai-analysis` 속성에 base64 형태로 실려 오버레이로 렌더된다.

근거:

- 식단 분석 속성: `js/app.js:6138-6143`
- 운동 분석 속성: `js/app.js:6157-6163`
- 수면 분석 속성: `js/app.js:6200-6205`

즉 현재 갤러리 공개는 “썸네일만 보인다” 수준이 아니라, 저장된 AI 분석 일부까지 UI 계층에서 다시 노출하는 구조다.

## 6. 신고 정책

## 6-1. 게시물 신고

게시물 신고는 클라이언트 `prompt()`로 사유를 받아 `reports/{reporterUid}_{docId}` 문서로 저장한다.

저장 필드:

- `reporterUid`
- `targetDocId`
- `targetUserId`
- `reason`
- `type: 'post'`
- `createdAt`

근거:

- `js/app.js:5240-5255`

특징:

- 같은 사용자가 같은 게시물을 다시 신고하면 문서 ID가 같아 덮어쓰기된다.
- 서버 자동 심사나 자동 숨김은 보이지 않는다.

## 6-2. 댓글 신고

댓글 신고는 `reports/{reporterUid}_{docId}_c{commentIdx}` 문서로 저장된다.

하지만 게시물 신고와 달리:

- `reason`이 없다
- `targetUserId`도 저장하지 않는다
- `commentIndex`만 저장한다

근거:

- `js/app.js:5287-5297`

즉 관리자 입장에서는 댓글 신고가 게시물 신고보다 문맥 정보가 더 빈약하다.

## 6-3. Firestore 규칙

`reports` 컬렉션은:

- 읽기: 관리자만
- 생성: 로그인했고 `reporterUid == auth.uid`
- 수정/삭제: 관리자만

근거:

- `firestore.rules:160-165`

즉 일반 사용자는 본인이 올린 신고도 다시 읽을 수 없고, 관리자 큐에 넣는 용도로만 쓰인다.

## 6-4. 관리자 후속 처리

관리자 화면에는 신고 목록 테이블이 있고 다음 두 액션이 있다.

- `resolveReport(id)`: 신고 문서 삭제
- `deletePost(docId, reportId)`: `daily_logs` 게시물 삭제 후 신고 문서도 삭제

근거:

- 신고 목록: `admin.html:1047-1067`
- 처리 완료: `admin.html:1069-1071`
- 게시물 삭제: `admin.html:1074-1080`

중요:

- 자동 숨김, 누적 신고 임계치, 사용자 정지 같은 정책은 코드상 보이지 않는다.
- 댓글 신고 전용 삭제 액션은 없다. 관리자 UI의 직접 삭제 버튼은 게시물(`daily_logs`) 단위다.

## 7. 차단 정책

## 7-1. 저장 방식

차단은 별도 컬렉션이 아니라 `users/{uid}.blockedUsers` 배열에 저장된다.

근거:

- 사용자 필드 허용: `firestore.rules:35-45`
- 로그인 시 로드: `js/auth.js:228`
- 로그아웃 시 초기화: `js/auth.js:326`

## 7-2. 실제 효과

차단 버튼을 누르면:

1. 내 `blockedUsers` 배열에 대상 uid를 추가
2. 로컬 `window._blockedUsers`를 갱신
3. 갤러리 정렬/필터 캐시를 다시 만들어 렌더

근거:

- `js/app.js:5264-5279`

실제 필터는 다음 한 줄이다.

- `cachedGalleryLogs` 중 `item.data.userId`가 차단 목록에 있으면 피드에서 제외

근거:

- `js/app.js:5555-5566`

즉 차단의 현재 의미는 “내 갤러리 피드에서 그 사람 게시물을 숨김”이다.

## 7-3. 차단의 한계

현재 코드 기준 차단은 다음을 하지 않는다.

- 서버에서 접근 차단
- 상대가 내 게시물을 못 보게 막기
- 상대의 댓글/반응을 전체 플랫폼에서 숨기기
- 직접 링크나 별도 읽기 경로 차단

특히 댓글 렌더 함수는 `blockedUsers`를 참고하지 않는다. 따라서 차단한 사용자의 게시물은 피드에서 사라져도, 그 사용자가 다른 사람 게시물에 남긴 댓글은 그대로 보일 수 있다.

근거:

- 댓글 렌더: `js/app.js:6260-6272`, `js/app.js:6077-6102`
- 차단 필터 적용 위치: `js/app.js:5555-5566`

## 7-4. 차단 해제 UI

현재 코드에서 차단 해제 UI는 보이지 않는다. `blockedUsers`에서 제거하는 전용 경로도 찾지 못했다.

근거:

- 차단 관련 검색 결과 기준 `blockUser` 추가만 존재

## 7-5. 관리자에서 보는 차단

관리자 화면은 차단 관계를 제재 로그처럼 다루지 않고, 각 사용자가 몇 명을 차단했는지 통계 형태로만 보여 준다.

근거:

- `admin.html:1083-1092`

## 8. 반응/댓글 정책과 실제 강제 수준

## 8-1. 앱이 의도한 사용 방식

클라이언트는 반응과 댓글을 비교적 제한적으로 다룬다.

- 반응: `arrayUnion` / `arrayRemove`
- 댓글: `arrayUnion`으로 추가, 본인 댓글만 `arrayRemove`

근거:

- 반응: `js/app.js:4461-4492`
- 댓글 추가/삭제: `js/app.js:6009-6063`

## 8-2. 하지만 Firestore 규칙은 더 느슨하다

`daily_logs` 업데이트 규칙은 서명된 사용자라면 변경 키가 `reactions`, `comments`뿐일 때 누구나 업데이트를 허용한다.

근거:

- `firestore.rules:82-88`

이 규칙은 앱의 정상 경로는 통과시키지만, 보안 관점에서는 다음을 함께 허용한다.

- 타인 게시물의 `comments` 전체 배열 교체
- 타인 게시물의 `reactions` 구조 전체 덮어쓰기
- 임의의 댓글 객체 삽입

즉 현재 정책은 “클라이언트가 선한 방식으로 쓴다”는 전제에 많이 기대고 있다.

## 8-3. 차단과 상호작용은 연결돼 있지 않다

차단 여부를 반응/댓글 쓰기 전에 검사하는 클라이언트 로직이나 보안 규칙은 보이지 않는다. 따라서 차단은 상호작용 금지보다 피드 숨김에 더 가깝다.

근거:

- 반응 함수: `js/app.js:4461-4492`
- 댓글 함수: `js/app.js:6009-6063`
- 차단 필터 위치: `js/app.js:5555-5566`

## 9. 구현상 주의점과 관찰

## 9-1. 공개 범위는 UI보다 넓다

갤러리 UI만 보면 사진과 텍스트 몇 개만 공개되는 것처럼 보이지만, 실제로는 `daily_logs` 문서 전체가 공개 읽기 대상이다. 건강 기록 앱 맥락에서는 민감도가 높은 편이다.

## 9-2. 공개 기간은 UI 기준 30일, 규칙 기준 무제한

현재 갤러리 화면은 최근 30일만 읽지만, 규칙은 `allow read: if true`라서 과거 문서도 별도 경로로 읽을 수 있다.

## 9-3. 신고는 수동 심사 중심이다

신고는 관리자 큐로 모이지만, 자동 숨김이나 자동 제재는 없다. 운영자가 직접 보고 삭제하는 구조다.

## 9-4. 차단은 개인 맞춤형 숨김이다

차단은 플랫폼 차원의 제재가 아니라 개인별 피드 필터다. 운영 정책 용어로는 “block”보다 “mute/hide”에 더 가깝다.

## 9-5. 댓글 신고 정보가 약하다

댓글 신고에는 사유와 대상 사용자 ID가 없어 운영자가 맥락을 재구성해야 한다.

## 10. 정리

해빛스쿨 갤러리는 커뮤니티 진입장벽을 낮추기 위해 의도적으로 공개 피드로 설계돼 있다. 다만 현재 구현은 “공개 피드 UX”와 “민감한 건강 기록 저장소”가 같은 `daily_logs` 문서를 공유하고 있어서, 노출 범위가 UI 인상보다 넓다.

운영·보안 관점에서 특히 중요한 포인트는 다섯 가지다.

1. 공개/비공개 토글 없이 기록 저장이 곧 갤러리 원본 생성이라는 점
2. `daily_logs` 전체 읽기가 비로그인 포함 모두에게 열려 있다는 점
3. 차단이 서버 강제가 아니라 개인 피드 숨김이라는 점
4. 신고가 자동 제재가 아닌 관리자 수동 처리라는 점
5. 댓글/반응 업데이트 권한이 규칙 레벨에서 꽤 넓게 열려 있다는 점

다음 단계로는 이 갤러리와 직접 맞닿는 `notifications` 도메인, 또는 운영 관점의 `admin` 자산/거래 이력 모델을 이어서 보면 전체 그림이 잘 이어진다.
