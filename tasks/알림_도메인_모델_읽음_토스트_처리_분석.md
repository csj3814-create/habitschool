# 알림 도메인 모델과 읽음/토스트 처리 분석

> 기준: 2026-04-03 정적 코드 추적
>
> 상태: 완료

## 요약

- 현재 알림 시스템은 하나가 아니라 두 채널로 나뉜다.
  - `notifications` 컬렉션 기반 인앱 알림
  - FCM 기반 브라우저 푸시 알림
- 인앱 알림은 Firestore 문서를 읽어 대시보드 진입 시 토스트로만 보여 주며, 별도 알림함 UI는 없다.
- 읽음 상태는 Firestore `read` 필드가 아니라 브라우저 `localStorage`의 마지막 확인 시각으로 관리한다.
- 반대로 FCM 푸시는 `users.fcmToken`을 써서 즉시 전송되지만, 대부분 `notifications` 컬렉션에 남지 않는다.
- 그래서 어떤 알림은 인앱 토스트만 있고, 어떤 알림은 OS 푸시만 있고, 두 채널이 완전히 동기화되지는 않는다.

## 1. 알림 채널은 둘로 분리돼 있다

### 1-1. Firestore `notifications`

- 친구 스트릭 달성, 소셜 챌린지 초대, 챌린지 시작, 챌린지 결산은 `notifications` 컬렉션 문서로 기록된다. `functions/index.js:870-878`, `functions/index.js:3116-3126`, `functions/index.js:3222-3230`, `functions/index.js:3364-3371`
- 클라이언트는 이 문서를 직접 읽어 토스트를 띄운다. `js/app.js:3053-3084`, `js/app.js:7519-7558`

### 1-2. FCM 푸시

- 일일 기록 리마인더, 스트릭 위기 알림, 관리자 전체 공지는 FCM으로 직접 보낸다. `functions/index.js:2854-2879`, `functions/index.js:2885-2913`, `functions/index.js:2919-2943`
- 포그라운드에서는 `onMessage`가 토스트로 보여 주고, 백그라운드에서는 서비스워커가 브라우저 알림을 띄운다. `js/auth.js:474-495`, `sw.js:22-31`

## 2. `notifications` 컬렉션 모델

### 2-1. 권한 규칙

- 읽기는 로그인 상태이면서 `postOwnerId == auth.uid`일 때만 허용된다. `firestore.rules:113-114`
- 생성은 로그인 상태라면 `postOwnerId`, `type`만 있고 자기 자신에게 보내는 것만 아니면 허용된다. `firestore.rules:115-117`
- 수정/삭제는 수신자 본인에게 허용된다. `firestore.rules:118`

### 2-2. 실제 저장 필드

- 현재 서버가 만드는 알림 문서는 공통적으로 `postOwnerId`, `type`, `createdAt`을 갖는다.
- 타입별로 `fromUserId`, `fromUserName`, `streakDays`, `challengeId`, `challengeType`, `durationDays`, `stakePoints`, `outcome`, `bonusPoints` 등이 붙는다. `functions/index.js:870-878`, `functions/index.js:3116-3126`, `functions/index.js:3222-3230`, `functions/index.js:3364-3371`
- `read`, `timestamp` 같은 필드는 현재 생산 코드에서 쓰이지 않는다.

### 2-3. 인덱스

- `postOwnerId + read + timestamp desc` 인덱스가 있다. `firestore.indexes.json:31-37`
- `postOwnerId + type + createdAt desc` 인덱스도 있다. `firestore.indexes.json:40-44`
- 실제 클라이언트 쿼리는 `postOwnerId + type + createdAt desc` 조합을 사용한다. `js/app.js:3060-3065`, `js/app.js:7524-7529`
- 반면 `read + timestamp` 인덱스는 현재 코드 경로상 사용 흔적이 없다.

## 3. 서버가 만드는 인앱 알림

### 3-1. 친구 스트릭 알림

- 사용자가 3, 7, 14, 30일 스트릭을 달성하면 친구들에게 `friend_streak` 알림을 생성한다. `functions/index.js:850-880`
- 중복 방지는 `daily_logs/{userId}_{date}.streakNotifiedDays` 배열로 한다. `functions/index.js:854-859`

### 3-2. 소셜 챌린지 알림

- 생성 시 초대 대상에게 `challenge_invite`를 보낸다. `functions/index.js:3114-3126`
- 전원 수락 후 활성화되면 참여자 전원에게 `challenge_started`를 보낸다. `functions/index.js:3220-3231`
- 결산 시 참여자 전원에게 `challenge_settled`를 보낸다. `functions/index.js:3362-3371`

## 4. 클라이언트가 실제로 소비하는 인앱 알림

### 4-1. 대시보드 진입 시 검사

- 대시보드 렌더 과정에서 `checkFriendStreakNotifications()`와 `checkChallengeNotifications()`가 호출된다. `js/app.js:2891-2898`
- 즉, 알림은 별도 실시간 리스너가 아니라 대시보드 진입 시점의 폴링에 가깝다.

### 4-2. 친구 스트릭 소비 로직

- `friendStreakNotifSeen_{uid}`를 `localStorage`에서 읽는다. `js/app.js:3056-3058`
- `notifications`에서 내 것 중 `friend_streak` 최근 5개를 읽는다. `js/app.js:3060-3065`
- `createdAt > lastSeen`인 문서만 새 알림으로 간주한다. `js/app.js:3068-3073`
- 실제 토스트는 가장 최근 1개만 띄우고, 이후 `localStorage`를 현재 시각으로 갱신한다. `js/app.js:3075-3080`

### 4-3. 챌린지 소비 로직

- `challengeNotifSeen_{uid}`를 `localStorage`에서 읽는다. `js/app.js:7521-7523`
- `challenge_invite`, `challenge_settled` 최근 5개를 읽는다. `js/app.js:7524-7529`
- 새 알림이면 각각 토스트를 띄우고, 하나라도 있으면 `localStorage`를 현재 시각으로 갱신한다. `js/app.js:7532-7554`
- 중요한 점은 서버가 만드는 `challenge_started`는 이 소비 로직에서 읽지 않는다.

## 5. 읽음 상태 모델

### 5-1. Firestore 읽음 필드를 쓰지 않는다

- 규칙과 인덱스만 보면 `read` 기반 모델 흔적이 있지만, 실제 생산 코드도 소비 코드도 `read`를 쓰지 않는다.
- `notifications` 문서를 읽은 뒤 `updateDoc(..., { read: true })` 같은 동작도 없다.

### 5-2. 실제 읽음 기준은 브라우저 로컬 시각

- 친구 스트릭과 챌린지 알림 모두 `localStorage`에 마지막 확인 시각을 밀리초 숫자로 저장한다. `js/app.js:3056-3058`, `js/app.js:7521-7523`
- 따라서 읽음 상태는 계정 공통이 아니라 “현재 브라우저/현재 기기 기준”이다.

## 6. FCM 푸시 모델

### 6-1. 토큰 등록

- 로그인 후 5초 뒤 부가 기능 로드 단계에서 `registerFCMToken(user)`가 호출된다. `js/auth.js:286-294`
- 브라우저 알림 권한을 요청하고, `firebase-messaging`을 동적 import한다. `js/auth.js:464-477`
- 토큰을 발급받아 `users.fcmToken`에 저장한다. `js/auth.js:479-489`
- `fcmToken`은 사용자 수정 허용 필드에 포함돼 있다. `firestore.rules:36-40`

### 6-2. 포그라운드/백그라운드 처리

- 포그라운드에서는 `onMessage()`가 `payload.data.title/body`를 그대로 `showToast()`에 전달한다. `js/auth.js:493-495`
- 백그라운드에서는 서비스워커가 `showNotification()`을 사용해 브라우저 푸시를 띄운다. `sw.js:22-31`
- 클릭 시 이미 열린 탭이 있으면 포커스만 주고, 없으면 `url`을 새 창으로 연다. `sw.js:96-113`

### 6-3. 푸시 생산자

- `sendDailyReminder`: 오늘 기록이 없는 사용자 대상 리마인더 푸시. `functions/index.js:2854-2879`
- `sendStreakAlert`: 스트릭이 있지만 오늘 기록하지 않은 사용자 대상 위기 푸시. `functions/index.js:2885-2913`
- `sendBroadcastNotification`: 관리자 전체 푸시. `functions/index.js:2919-2943`, `admin.html:1238-1248`
- 이 푸시들은 `notifications` 컬렉션에는 별도로 남지 않는다.

## 7. 구현상 주의점

### 7-1. 생산자와 소비자 타입이 완전히 맞지 않는다

- 서버는 `challenge_started`를 만들지만, 클라이언트 조회는 `challenge_invite`, `challenge_settled`만 본다. `functions/index.js:3222-3230`, `js/app.js:7524-7529`
- 그래서 “챌린지가 시작됐다”는 인앱 알림 문서는 쌓여도 현재 토스트에는 안 뜬다.

### 7-2. 읽음 상태가 계정 단위로 동기화되지 않는다

- Firestore `read` 필드를 안 쓰고 브라우저 `localStorage`만 쓰기 때문에 다른 기기에서는 같은 알림이 다시 뜰 수 있다.
- 반대로 로컬스토리지를 지우면 예전 알림도 다시 새 알림처럼 보일 수 있다.

### 7-3. 최근 5개 제한 때문에 오래된 미확인 알림이 누락될 수 있다

- 친구 스트릭과 챌린지 알림 모두 최근 5개만 읽는다. `js/app.js:3064-3065`, `js/app.js:7528-7529`
- 앱을 오래 안 열었거나 짧은 시간에 알림이 많이 쌓이면, 최신 5개 바깥의 미확인 알림은 토스트 기회를 잃는다.

### 7-4. 친구 스트릭 로직은 가장 최근 1개만 토스트로 보여 준다

- 새 친구 스트릭 알림이 여러 개 있어도 첫 번째 1개만 보여 준다. `js/app.js:3077-3079`

### 7-5. 보안 규칙이 실제 의도보다 넓다

- 현재 코드는 Cloud Functions만 `notifications`를 만드는 구조처럼 보이지만, 규칙상 아무 로그인 사용자나 임의의 `postOwnerId`, `type`으로 알림 문서를 만들 수 있다. `firestore.rules:113-118`
- 수신자 본인은 문서를 수정/삭제할 수 있어서, 원본 보존형 이벤트 로그라기보다 개인 inbox 문서에 더 가깝다.

### 7-6. 인덱스와 실제 모델 사이에 레거시 흔적이 남아 있다

- `read + timestamp` 인덱스가 있지만 현재 문서는 `createdAt`만 주로 쓰고 `read`와 `timestamp`는 안 쓴다. `firestore.indexes.json:31-44`, `functions/index.js:870-878`, `functions/index.js:3116-3126`

### 7-7. 푸시와 인앱 알림이 서로의 이력을 보존하지 않는다

- 리마인더/스트릭 위기/관리자 공지는 푸시만 있고 `notifications` 문서가 없다.
- 반대로 친구 스트릭/소셜 챌린지는 인앱 문서만 있고 OS 푸시는 없다.
- 결과적으로 사용자는 알림 채널별로 경험이 분리된다.

## 8. 모델 해석

- `notifications`: 대시보드 진입 시 읽는 인앱 토스트 소스
- `localStorage`: 기기별 임시 읽음 체크포인트
- `users.fcmToken`: 브라우저 푸시 전달 주소
- FCM: 즉시성 높은 리마인더/공지 채널
- 서비스워커: 백그라운드 푸시 표시 계층

## 결론

- 현재 알림 도메인은 “단일 알림 시스템”이 아니라 “인앱 문서형 토스트”와 “FCM 푸시”가 병렬 운영되는 이중 구조다.
- 읽음 상태는 서버 동기화형 모델이 아니라 기기 로컬 기준이어서, 알림함이나 다기기 일관성보다는 단순 토스트 UX에 맞춰져 있다.
- 이후 개선 작업을 한다면 우선 아래 세 가지를 먼저 결정해야 한다.
  - `notifications`를 실제 inbox 원장으로 볼지
  - `read`를 Firestore 기준으로 복구할지
  - FCM 푸시와 인앱 알림을 같은 이벤트 소스에서 파생시킬지
