## 1. 요약

해빛스쿨의 주간 미션은 별도 컬렉션이 아니라 `users/{uid}` 문서 안의 상태로 관리된다.

핵심 상태 필드:

- `missionLevel`
- `weeklyMissionData`
- `missionHistory`
- `missionStreak`
- `missionBadges`
- `selectedMissions` (레거시)

상태 전이는 크게 아래 흐름이다.

1. 미션 없음
2. 이번 주 미션 설정
3. 주간 진행 중
4. 주차가 바뀌면 이전 주 미션을 백그라운드 아카이브
5. 새 주는 다시 “미션 없음” 상태로 시작
6. 필요하면 같은 주 안에서도 재설정 가능
7. 모든 미션 완료 시 레벨업 가능

핵심 근거 코드:

- 미션 상태 읽기/렌더: `js/app.js:2562-2885`
- 아카이브/리셋: `js/app.js:3234-3314`
- 저장/재설정/레벨업: `js/app.js:3318-3434`
- 미션 정의/배지/주차 계산: `js/firebase-config.js:41-210`
- Firestore 허용 필드: `firestore.rules:42-56`

## 2. 상태 저장 위치와 데이터 모델

## 2-1. `users` 문서 안에서 관리된다

주간 미션은 `users/{uid}`에 아래 필드들로 저장된다.

- `missionLevel: number`
- `weeklyMissionData: { weekId, missions[] } | null`
- `missionHistory: array`
- `missionStreak: number`
- `missionBadges: string[]`
- `selectedMissions: string[]` (레거시)

근거:

- `js/app.js:2568-2574`
- `firestore.rules:42-56`

## 2-2. `weeklyMissionData` 구조

현재 주의 활성 미션은 아래 형태다.

```js
{
  weekId: "2026-W14",
  missions: [
    {
      id,
      text,
      target,
      type,        // diet | exercise | mind
      difficulty,  // preset만 존재
      isCustom     // custom이면 true
    }
  ]
}
```

근거:

- `js/app.js:3370-3374`

## 2-3. `missionHistory` 구조

주차가 끝나면 아래 정보가 히스토리에 들어간다.

```js
{
  weekId,
  missions,
  stats: { diet, exercise, mind },
  completionRate
}
```

보관 개수는 최근 12주다.

근거:

- `js/app.js:3266-3272`

## 3. 시간 기준과 주차 계산

## 3-1. 날짜 기준은 KST

주간 미션은 한국 시간 기준으로 동작한다.

- `getDatesInfo()`는 KST 기준 `todayStr`, `weekStrs`를 만든다.
- `weekStrs`는 월요일~일요일 7일 문자열 배열이다.

근거:

- `js/ui-helpers.js:4-26`

## 3-2. `weekId`는 ISO 주차

`getWeekId(dateStr)`는 ISO 8601 방식으로 주차를 계산한다.

- 월요일 시작
- 연도 경계도 ISO 규칙으로 처리

근거:

- `js/firebase-config.js:197-203`

## 4. 미션이 없는 상태

근거:

## 5. 미션 설정 흐름

## 5-1. 프리셋 미션

프리셋은 레벨별 `MISSIONS[level]` 정의에서 가져온다.

- 레벨은 1~5
- 각 레벨마다 `diet`, `exercise`, `mind`
- 각 카테고리마다 `easy`, `normal`, `hard`

초기 선택은 각 카테고리 `normal`이다.

근거:

- `js/firebase-config.js:88-183`
- `js/app.js:2912-2921`
- `js/app.js:3175-3188`

## 5-2. 커스텀 미션

커스텀 미션은 최대 5개까지 임시 배열 `pendingCustomMissions`에 쌓인다.

입력 필드:

- `type`: `diet` / `exercise` / `mind`
- `text`: 30자 이내 자유 문구
- `target`: 1~7일

근거:

- `js/app.js:3190-3213`

중요한 해석:

커스텀 미션은 텍스트가 자유롭지만, 판정 로직은 자유롭지 않다. 실제 달성 판정은 `type`에 따라 해당 카테고리의 주간 일수 카운터를 그대로 쓴다.

예를 들어:

- `🥗 물 2L 마시기`를 `diet` 타입으로 만들면
- 실제로는 “식단 기록 인정 일수”가 목표치에 도달했는지로 판정된다.

즉 현재 구현의 커스텀 미션은 “자유 문구 라벨이 붙은 카테고리형 목표”에 가깝다.

근거:

- 커스텀 생성: `js/app.js:3194-3210`
- 진행 계산: `js/app.js:2792-2808`

## 5-3. 저장 시점

`saveWeeklyMissions()`는 다음을 수행한다.

1. 현재 KST 기준 `currentWeekId` 계산
2. 체크된 프리셋 미션 수집
3. 커스텀 미션 합치기
4. 미션이 1개 이상인지 확인
5. `firstMission` 배지 최초 부여
6. `weeklyMissionData`를 현재 주차로 저장

근거:

- `js/app.js:3318-3387`

특이점:

- 저장 시 `selectedMissions`는 더 이상 메인 저장 경로가 아니다.
- 현재 구현의 표준 상태는 `weeklyMissionData`다.

근거:

- `js/app.js:2574-2605`

## 7. 진행 중 상태

현재 주 `weeklyMissionData.weekId === currentWeekId`이고 미션 배열이 있으면 “진행 중 모드”가 된다.

근거:

- `js/app.js:2629`

## 7-1. 진행도 원천 데이터

주간 진행도는 `weekLogs`의 `awardedPoints`에서 카테고리별 “일수”를 센다.

- `diet` true인 날짜 수
- `exercise` true인 날짜 수
- `mind` true인 날짜 수

근거:

- `js/app.js:2645-2651`

즉 주간 미션은 “주간 누적 포인트”가 아니라 “카테고리별 달성 일수” 기반이다.

## 7-2. 개별 미션 판정

각 미션은 자신의 `type`에 해당하는 카운터만 본다.

- `diet` 미션이면 `statDiet`
- `exercise` 미션이면 `statExer`
- `mind` 미션이면 `statMind`

이 값과 `target`을 비교해 개별 진행 퍼센트를 계산한다.

근거:

- `js/app.js:2792-2808`

## 7-3. 진행 중 화면의 전체 달성률

진행 중 화면의 `overallRate`는 “미션 평균” 방식이다.

- 각 미션별 `min(val / target, 1)` 계산
- 그 평균을 퍼센트로 환산

즉 목표치가 큰 미션과 작은 미션을 동일 가중치로 취급한다.

근거:

- `js/app.js:2819-2824`

## 7-4. 완료 판단과 버튼 변화

- 모든 미션이 100%면 `completedMissions === totalMissions`
- 이때 레벨 5 미만이면 레벨업 버튼 표시
- 완료 전에는 저장 버튼을 “미션 진행 중...” 상태로 비활성화
- 진행 중에도 “이번 주 미션 재설정” 버튼은 보인다

근거:

- `js/app.js:2845-2867`

## 8. 재설정 흐름

사용자는 같은 주 안에서도 미션을 리셋할 수 있다.

`resetWeeklyMissions()`는:

- `weeklyMissionData: null`

중요한 점:

- `missionHistory`, `missionStreak`, `missionBadges`는 건드리지 않는다.
- 주간 기록(`daily_logs`)도 그대로 남는다.

즉 재설정은 “이번 주 활성 미션 설정만 지우는 동작”이다.

근거:

- `js/app.js:3390-3401`

레벨업은 UI에서만 노출된다.

조건:

## 10. 주차 전환과 아카이브 상태 전이

## 10-1. 전환 감지

대시보드 렌더 시점에:

- `weeklyMissionData.weekId`가 있고
- 현재 `currentWeekId`와 다르면
- 지난주 데이터로 판단한다

근거:

- `js/app.js:2607-2608`

## 10-2. 즉시 화면 상태

주차가 바뀐 것으로 감지되면 렌더 함수는:

- `archiveWeekAndReset(...)`를 백그라운드로 호출
- 로컬 변수 `weeklyMissionData = null`
- 로컬 변수 `missionStreak = 0`

으로 처리한 뒤 새 주의 “미션 없음” 상태로 그린다.

근거:

- `js/app.js:2612-2626`

해석:

주차가 바뀐 첫 진입 시에는 아카이브가 끝나기 전에도 새 주 미션 선택 화면이 먼저 열린다.

## 10-3. 아카이브 대상 기간

아카이브 함수는 현재 주 `weekStrs`에서 7일을 뺀 `prevWeekStrs`를 만들어 지난주 월~일 범위를 구한다.

근거:

- `js/app.js:2615-2619`
- `js/app.js:3234-3240`

## 10-4. 아카이브 계산

`archiveWeekAndReset()`는 지난주 `daily_logs`를 읽어:

- `statDiet`
- `statExer`
- `statMind`

를 구한 뒤, 저장된 `weeklyData.missions`를 기준으로 `completionRate`를 계산한다.

계산식:

- `totalTarget += mission.target`
- `totalAchieved += min(actual, target)`
- `completionRate = totalAchieved / totalTarget`

즉 아카이브용 완료율은 “목표 일수 가중치” 방식이다.

근거:

- `js/app.js:3241-3260`

## 10-5. 스트릭 규칙

주간 스트릭은 지난주 완료율이 80% 이상이면 이어지고, 아니면 0으로 초기화된다.

```js
newStreak = completionRate >= 80 ? currentStreak + 1 : 0
```

근거:

- `js/app.js:3262-3263`

## 10-6. 히스토리 보관

아카이브 엔트리는 `missionHistory`에 append되고 최근 12주만 유지한다.

근거:

- `js/app.js:3266-3272`

## 12. 레이스 방지 장치

## 12-2. 새 주 미션 덮어쓰기 방지

`archiveWeekAndReset()`는 Firestore 쓰기 직전에 다시 `users/{uid}`를 읽고, 현재 `weeklyMissionData.weekId`를 확인한다.

- 이미 새 주 미션이 저장됐으면 `weeklyMissionData: null` 덮어쓰기를 생략

근거:

- `js/app.js:3289-3304`

이 로직이 바로 “주차 전환 직후 사용자가 먼저 새 미션을 저장했는데, 뒤늦은 archive가 null로 덮어쓰는 문제”를 막는 장치다.

## 13. 현재 구현에서 중요한 차이와 주의점

## 13-1. 진행 중 달성률과 아카이브 완료율은 계산 방식이 다르다

진행 중 화면 `overallRate`:

- 미션별 평균
- 모든 미션이 동일 가중치

주차 종료 후 `completionRate`:

- 목표 일수 합 기준
- 목표치가 큰 미션이 더 큰 비중

즉 같은 주를 두고도:

- 진행 중 UI의 “이번 주 달성률”
- 지난주 결과 카드와 스트릭 판정에 쓰이는 “완료율”

이 서로 다를 수 있다.

근거:

- 진행 중 계산: `js/app.js:2819-2824`
- 아카이브 계산: `js/app.js:3254-3260`

## 13-2. 커스텀 미션은 자유 텍스트지만 판정은 카테고리 기반이다

텍스트는 자유롭게 쓰지만 실제 판정은 `diet/exercise/mind` 카운터만 본다.

즉 현재 구조의 커스텀 미션은 “새로운 측정값”을 만드는 기능이 아니다.

근거:

- 생성: `js/app.js:3194-3210`
- 판정: `js/app.js:2792-2808`

## 13-3. 주차가 바뀌는 첫 렌더는 즉시 새 주 화면으로 간다

현재 렌더 경로상 아카이브는 백그라운드 작업이고, 렌더는 먼저 `weeklyMissionData = null`, `missionStreak = 0`로 바뀐다.

추론:

- 주차 경계 첫 진입 시, 아카이브가 서버에 새 `missionHistory`와 `missionStreak`를 저장하더라도
- 같은 렌더 사이클 안에서는 그 값이 다시 반영되지 않는다
- 따라서 첫 진입 화면은 잠시 보수적으로 초기 상태처럼 보일 수 있다

이건 코드 흐름으로부터의 추론이다.

근거:

- `js/app.js:2612-2626`
- archive 성공 후 별도 `renderDashboard()` 재호출 없음

## 13-4. 캐시 무효화는 저장 경로마다 일관되지 않다

`saveWeeklyMissions()`는 `_invalidateDashboardCache()`를 호출한 뒤 `renderDashboard()`를 다시 그린다. 반면 `resetWeeklyMissions()`와 `levelUp()`은 같은 함수 호출이 없다.

추론:

- 메모리/localStorage 캐시 TTL 안에서는 리셋/레벨업 직후 화면이 잠깐 이전 상태를 보여줄 가능성이 있다

이 역시 코드 흐름 기반 추론이다.

근거:

- 저장 시 캐시 무효화: `js/app.js:3380-3383`
- 리셋: `js/app.js:3390-3401`
- 레벨업: `js/app.js:3426-3434`

## 14. 결론

해빛스쿨의 주간 미션은 “현재 주 활성 상태”와 “지난주 히스토리/스트릭/배지”를 같은 `users` 문서 안에서 관리하는 클라이언트 중심 상태 머신이다.

핵심적으로 기억할 점은 네 가지다.

1. 활성 상태의 기준은 `weeklyMissionData.weekId === currentWeekId`다.
2. 주차가 바뀌면 렌더 시점에 백그라운드 아카이브가 돌고, 새 주는 즉시 미션 선택 상태로 전환된다.
3. 스트릭과 배지는 아카이브 시점의 `completionRate` 80%/100% 규칙으로 결정된다.
4. 진행 중 달성률과 주차 종료 완료율의 계산식, 그리고 커스텀 미션의 실제 판정 방식은 UI 문구만 보고는 오해하기 쉬운 부분이다.
