## 이번에 확정한 제품 규칙

- 친구 연결은 `요청 -> 수락/거절 -> active` 흐름으로만 만든다.
- 친구 관계는 `쌍(pair)` 단위다.
- A-B가 친구가 되어도 A-C, B-C는 자동 연결되지 않는다.
- 전체 친구 수는 제한하지 않는다.
- `1:1 경쟁`은 총 2인이다.
- `함께 목표`는 총 3인이다.
- `함께 목표`의 3인은 `생성자 1명 + 초대 2명` 기준이다.
- 그룹 챌린지는 `모든 참가자끼리 서로 친구`까지는 요구하지 않는다.
- 그룹 챌린지는 `생성자와 각 참가자가 active friendship`이면 충분하다.
- 친구 요청은 3일 뒤 만료된다.
- 대시보드/갤러리는 전체 친구 목록이 아니라 `오늘 활동한 친구` 중심으로 보여준다.

## 목표

- 해빛코치 `!등록 코드`로 앱 계정과 카카오 사용자를 1회 연결한다.
- 해빛코치 `!친구 코드`는 즉시 친구 추가가 아니라 `친구 요청 생성`만 한다.
- 친구 연결은 반드시 해빛스쿨 앱에서 `수락` 또는 `거절`로 확정한다.
- 기존 단방향 `friends` 데이터는 이번에 리셋하고 새 흐름으로 다시 받는다.
- 소셜 챌린지는 `active friendship` 기준으로 허용한다.

## 1회성 리셋 / 마이그레이션

이번 릴리즈에서는 기존 단방향 친구를 살리지 않고 리셋한다.

### 필수 작업

1. 모든 `users/{uid}.friends`를 `[]`로 초기화
2. 기존 `friendships` 문서가 있으면 삭제하거나 `removed` 처리
3. 테스트/구버전 `friend_request`, `friend_connected` 알림이 있으면 함께 정리

### 이유

- 예전 단방향 친구가 남아 있으면 새 요청 흐름과 데이터 해석이 꼬일 수 있다.
- 이번부터는 `friendships/{pairId}`가 원본이다.
- 소셜 챌린지 조건도 `active friendship` 기준으로 함께 맞추는 편이 안전하다.

## 2. `functions/index.js`

### A. 새 callable 추가

권장 함수 3개:

1. `requestFriend`
2. `respondFriendRequest`
3. `removeFriendship`

#### `requestFriend`

입력 예시:

```json
{
  "friendCode": "ABC123",
  "source": "app"
}
```

동작:

- 현재 사용자 확인
- 자신의 `referralCode`와 동일하면 거절
- `friendCode`로 대상 사용자 조회
- `pairId` 계산
- 기존 `friendships/{pairId}` 조회
- 이미 `active`면 종료
- 기존 `pending`이고 만료 안 됐으면 상태에 맞는 결과 반환
- 새 요청이면 `status: 'pending'`
- `expiresAt = now + 3일`
- 대상 사용자에게 `friend_request` 알림 생성

#### `respondFriendRequest`

입력 예시:

```json
{
  "friendshipId": "uidA__uidB",
  "accept": true
}
```

동작:

- 현재 사용자가 `pendingForUid`인지 확인
- 만료면 `expired`
- 수락이면
  - `status: 'active'`
  - `respondedAt`, `acceptedAt` 기록
  - 양쪽 `users/{uid}.friends`에 서로 `arrayUnion`
  - 양쪽에 `friend_connected` 알림 생성
- 거절이면
  - `status: 'declined'`
  - `respondedAt` 기록
  - 선택적으로 요청자에게 `friend_declined` 알림 생성

#### `removeFriendship`

입력 예시:

```json
{
  "friendshipId": "uidA__uidB"
}
```

동작:

- 현재 사용자가 당사자인지 확인
- `status: 'removed'`
- 양쪽 `users/{uid}.friends`에서 서로 제거

### 권장 위치

### 권장 동작

1. 로그인 사용자 기준 새 연결 코드 생성
2. `users/{uid}`에 아래 필드 저장

3. 화면에 코드 표시
4. `복사` 버튼 제공
5. 안내 문구 표시
   - `카카오톡 방에서 !등록 ABCD1234 입력`
   - `10분 뒤 만료`

### 권장 구현 방식

가장 안전한 방식은 callable `generateChatbotLinkCode`를 두는 것이다.

이유:

- 중복 코드 충돌을 서버에서 막기 쉽다.
- 만료 시각을 서버 기준으로 넣을 수 있다.
- 클라이언트가 임의 값을 쓰지 않게 제어할 수 있다.

## Firestore Rules / 인덱스 메모

### Rules

- `friendships`는 클라이언트 직접 쓰기보다 callable 전용이 안전하다.
- `users`에 새 필드를 쓰는 경우 rules whitelist 반영 필요
  - `friends`
  - `chatbotLinkCode`
  - `chatbotLinkCodeExpiresAt`
  - `chatbotLinkCodeGeneratedAt`
  - `chatbotLinkCodeLastUsedAt`는 서버/관리 경로만 허용 권장

### 인덱스

알림 조회가 늘어나면 아래 조합이 필요할 수 있다.

- `notifications(postOwnerId, type, createdAt desc)`

`friendships` 목록/함수에서 복합 조회를 넣는다면 필요한 인덱스를 같이 확인한다.

3. 챗봇이 연결 성공 응답
4. Firestore에서 코드 필드가 제거되고 `chatbotLinkCodeLastUsedAt`만 남는지 확인

### 2. 친구 요청/수락

1. A가 `!내코드`
2. B가 `!친구 A코드`
3. 앱에서 A에게 `friend_request` 알림 노출
4. A가 앱에서 수락
5. `friendships/{A__B}.status === 'active'`
6. A/B 양쪽 `users.friends`에 서로 uid 반영

### 3. 그룹 친구 구조

1. A-B 친구 연결
2. A-C 친구 연결
3. B-C는 요청/수락 전까지 친구 아님
4. 이 상태에서도 A가 B, C를 함께 목표 챌린지에 초대할 수 있어야 함

### 4. 챌린지 인원

1. `1:1 경쟁`
   - 친구 1명만 선택 가능
2. `함께 목표`
   - 친구 최대 2명만 선택 가능
   - 총 참가자 수는 3명

### 5. 만료

1. 친구 요청 생성
2. 3일 뒤 수락 시도
3. `expired` 처리되고 친구 연결이 되지 않는지 확인

### 6. 오늘 활동 친구 카드

1. 친구가 여러 명 있어도 오늘 기록한 친구만 노출
2. 기록이 없는 친구는 대시보드/갤러리 상단 카드에서 제외

## 이번 단계의 비목표

- 기존 단방향 친구 데이터 보존
- 챗봇 안에서 친구 수락 처리
- `referralCode`를 `friendCode`로 완전히 분리

위 3개는 이번 릴리즈 범위에서 제외한다.
