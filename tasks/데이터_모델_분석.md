# 해빛스쿨 데이터 모델 분석

작성일: 2026-04-03  
분석 방식: 코드 정적 분석  
분석 범위: Firestore 중심 데이터 구조, 문서 ID 규칙, 주요 필드, 읽기/쓰기 주체, 컬렉션 간 관계

---

## 1. 요약

해빛스쿨의 데이터 모델은 크게 네 층으로 나뉜다.

1. 사용자 상태
2. 일일 활동 로그
3. 보상/온체인/챌린지 로그
4. 운영/통계/관리 데이터

핵심 구조는 아래 4개 컬렉션이 중심이다.

- `users`
- `daily_logs`
- `blockchain_transactions`
- `social_challenges`

그리고 이들을 보조하는 파생/운영 컬렉션이 붙는다.

- `notifications`
- `meta/communityStats`
- `communityStats_archive`
- `monthly_rewards`
- `reports`
- `emailLogs`
- `mining_rate_history`
- `pointAdjustments`
- `admins`

핵심 규칙 파일:

- `firestore.rules`

주요 근거 코드:

- 클라이언트 저장/조회: `js/app.js`, `js/auth.js`, `js/blockchain-manager.js`
- 서버 저장/집계: `functions/index.js`
- 운영자 화면: `admin.html`

## 2. 큰 구조 이해

### 2-1. `users`는 사용자 현재 상태 문서다

이 컬렉션은 “현재 상태”를 담는다.

- 현재 포인트
- 표시용 이름
- 건강 프로필
- 추천 코드
- 친구 목록
- 미션 상태
- 개인 챌린지 진행 상태
- 차단 목록

즉, `users`는 스냅샷이고, 이력성 데이터는 별도 컬렉션이나 서브컬렉션으로 빠진다.

### 2-2. `daily_logs`는 하루 단위 원장이다

이 컬렉션은 사실상 앱의 핵심 원장이다.

- 하루 기록 본문
- 포인트 계산 근거
- 갤러리 게시물 원본
- 댓글/리액션 대상
- 스트릭 계산 원본
- 주간/월간 집계의 원본

중요한 해석:

`daily_logs`는 “개인 일지”이면서 동시에 “공개 커뮤니티 포스트”다.

### 2-3. `blockchain_transactions`는 온체인/보상 이력 로그다

사용자 상태가 아니라 사건(event) 로그다.

- 포인트→HBT 변환
- 스테이킹
- 개인 챌린지 정산
- HBT 마이그레이션

이 컬렉션은 자산 화면, 운영자 화면, 통계 화면이 함께 본다.

### 2-4. `social_challenges`는 친구 간 협동/경쟁 상태 머신이다

단순 초대 목록이 아니라 상태가 변하는 별도 도메인 모델이다.

- `pending`
- `active`
- `settled`
- `cancelled`

## 6. 컬렉션 간 관계

## 6-1. 사용자 기준 관계

- `users/{uid}`
  - 1:N `daily_logs`
  - 1:N `blockchain_transactions`
  - 1:N `notifications` 수신자 관점
  - 1:N `users/{uid}/inbodyHistory`
  - 1:N `users/{uid}/bloodTests`

## 6-2. 로그 기준 관계

- `daily_logs`
  - 포인트의 원본
  - 스트릭 계산의 원본
  - 갤러리 게시물의 원본
  - 월간 MVP/커뮤니티 통계의 원본

## 6-3. 챌린지 기준 관계

- 개인 챌린지
  - 상태는 `users.activeChallenges`
  - 이력은 `blockchain_transactions`

- 소셜 챌린지
  - 상태는 `social_challenges`
  - 알림은 `notifications`
  - 포인트 결과는 다시 `users.coins`

## 7. 쓰기 흐름 기준으로 본 수명주기

## 7-1. 신규 사용자

1. 로그인
2. `users/{uid}` 생성 또는 병합
3. 지갑 초기화 시 `walletAddress`, `referralCode` 등 추가
4. 온보딩 완료 시 `onboardingComplete`
5. 환영 보너스 지급 시 `coins`, `welcomeBonusGiven`

## 7-2. 하루 기록 저장

1. 클라이언트가 `daily_logs/{uid}_{date}` 저장
2. 서버가 `awardedPoints` 차이만큼 `users.coins` 증가
3. 서버가 `currentStreak`, 추천 보상, 친구 알림 반영
4. 사용자는 대시보드/자산/갤러리에서 결과 확인

## 7-3. 포인트와 HBT

1. 사용자는 포인트를 쌓는다
2. 변환 시 `blockchain_transactions`에 `conversion`
3. 개인 챌린지 시작 시 `users.activeChallenges`와 `blockchain_transactions/staking`
4. 정산 시 `blockchain_transactions/challenge_settlement`

## 7-4. 커뮤니티

1. `daily_logs`가 공개 피드가 된다
2. 반응/댓글은 같은 문서에 인라인 저장된다
3. 서버가 반응 포인트를 다시 `users.coins`에 반영한다

## 7-5. 소셜 챌린지

1. `social_challenges` 생성
2. `notifications`에 초대 생성
3. 모두 수락 시 `active`
4. 스케줄러가 정산
5. `users.coins` 지급 + `notifications` 정산 알림

## 8. 설계상 주의할 점

## 8-1. `daily_logs`는 공개 컬렉션이다

읽기 권한이 전체 공개라서, 여기에 들어가는 데이터는 사실상 공개 게시물 데이터로 취급해야 한다. 건강 수치와 공개 피드 데이터가 섞여 있다는 점은 설계상 민감하다.

## 8-2. `users`는 “현재 상태”가 많이 응집된 비대한 문서다

닉네임, 건강 프로필, 추천, 미션, 지갑, 포인트, 차단 목록, 관리자 피드백이 한 문서에 모인다. 추후 필드 충돌이나 병합 규칙을 주의해야 한다.

## 8-3. 서버 전용 필드가 규칙 화이트리스트보다 넓다

`createdAt`, `welcomeBonusGiven`, `referredBy`, `lastGasFunded`, `adminFeedback` 같은 필드는 서버가 쓰고, 클라이언트 수정 가능 필드 목록과는 별개다.  
즉 “규칙 문서만 보고 전체 스키마를 판단하면 빠지는 필드가 있다.”

## 8-4. 레거시 구조 흔적이 남아 있다

- `activeChallenge`와 `activeChallenges` 혼재
- `hbtBalance` 과거 흔적
- `health_profiles`, `reactions`, `admin_feedback`, `admin_messages` 규칙 잔존
- 운동 필드의 레거시/신규 구조 공존

## 8-5. 문서 ID 규칙은 꽤 중요하다

- `users`: UID
- `daily_logs`: `uid_date`
- `communityStats_archive`: `YYYY-MM`
- `monthly_rewards`: `YYYY-MM`
- `bloodTests`, `inbodyHistory`: 날짜 기반

이 규칙을 이해하면 디버깅 속도가 크게 빨라진다.
