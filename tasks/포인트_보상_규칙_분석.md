## 1. 요약

해빛스쿨의 보상 시스템은 크게 세 층으로 나뉜다.

1. 사용자가 하루 기록을 저장할 때 프런트엔드가 `daily_logs.awardedPoints`를 계산한다.
2. Firestore 트리거와 callable function이 `users.coins`를 실제로 증감시킨다.
3. 별도 규칙으로 추천, 마일스톤, 리액션, 월간 MVP, 개인 챌린지, 소셜 챌린지, HBT 변환이 붙는다.

핵심 해석:

- “포인트 계산”과 “포인트 지급”은 같은 로직이 아니다.
- 일일 기록 기본 포인트는 클라이언트가 계산하지만, 실제 `coins` 증가는 서버가 확정한다.
- 챌린지와 소셜 기능은 “추가 포인트 지급”뿐 아니라 “포인트/HBT 잠금, 반환, 소각”까지 포함하는 구조다.

핵심 근거 코드:

## 4. 기본 포인트가 실제 지급되는 방식

### 4-1. 프런트는 계산만 한다

클라이언트는 저장 직전 `awardedPoints.dietPoints`, `exercisePoints`, `mindPoints`를 갱신하고, 화면에는 `pointsToGive`만큼 즉시 증가한 것처럼 보여준다.

근거:

- `js/app.js:4358-4367`
- `js/app.js:4405-4408`

### 4-2. 서버가 `users.coins`를 확정한다

실제 포인트 지급은 `daily_logs` 변경 트리거 `awardPoints`가 담당한다.

- 새 합계와 이전 합계의 차이(`diff`)를 계산
- `diff > 0`일 때만 `users/{uid}.coins += diff`
- 이후 스트릭 계산, 추천 보상 체크, 친구 알림까지 같은 트리거 안에서 수행

근거:

- `functions/index.js:780-823`

### 4-3. 중요한 구현 특성: 차감은 없다

클라이언트는 `pointsToGive`를 카테고리별 `Math.max(0, new - old)`로 계산한다. 서버도 `diff <= 0`이면 종료한다.

즉 현재 구현 기준으로는:

- 이미 받은 일일 기본 포인트는 같은 날짜 편집으로 줄어들어도 회수되지 않는다.
- 포인트는 “증가분만 반영”되는 누적형이다.

근거:

- `js/app.js:4358-4360`
- `functions/index.js:792-794`

이건 버그라고 단정할 수는 없지만, 운영 규칙을 설명할 때는 반드시 명시해야 하는 포인트다.

## 8. 소셜 챌린지 보상

소셜 챌린지는 별도 `social_challenges` 도메인에서 처리되며, 포인트 이동이 직접 발생한다.

## 8-1. 단체 목표형 (`group_goal`)

핵심 규칙:

- 참가자 모두가 `durationDays` 대비 70% 이상 활동일을 채워야 성공
- 성공 시 각자 `habitPoints`의 20%를 추가 포인트로 받는다
- 보너스는 `Math.floor`로 내림 처리

여기서 `habitPoints`는 기간 내 `daily_logs.awardedPoints` 합계다.

근거:

- 기준 상수: `functions/index.js:2991`
- 집계/지급: `functions/index.js:3245-3302`

## 8-2. 1:1 경쟁형 (`competition`)

시작 시점:

- 생성자 스테이크 포인트 차감
- 수락자도 수락 시 같은 스테이크 포인트 차감
- 초대 만료나 전원 거절이면 생성자 스테이크 반환

근거:

- 생성자 락업: `functions/index.js:3084-3088`

- 만료/거절 반환: `functions/index.js:3158-3182`, `functions/index.js:3389-3397`

결산 시점:

- 한쪽이 활동일 0일이면 무효, 양쪽 모두 전액 반환
- 동점이면 양쪽 모두 전액 반환
- 승부가 나면 승자에게:
  - 자기 스테이크 반환
  - 상대 스테이크 획득
  - 자기 `habitPoints`의 30% 보너스 추가

근거:

- 무효/동점 반환: `functions/index.js:3306-3333`
- 승자 지급: `functions/index.js:3335-3349`
- 보너스 비율: `functions/index.js:2992`, `functions/index.js:3339`

## 8-3. 구현상 해석

소셜 챌린지는 “기본 포인트를 새로 계산해 더 준다”기보다:

- 이미 얻은 기본 포인트를 집계 기준으로 삼고
- 그 결과에 따라 포인트를 추가 보너스로 주거나
- 경쟁 스테이크를 재분배하는 구조다.

## 9. 포인트→HBT 변환 규칙

## 9-1. 서버가 강제하는 실제 규칙

- 최소 100P
- 100P 단위만 허용
- 사용자의 현재 `coins`를 서버가 재검증
- 일일 총 HBT 변환 한도는 12000 HBT
- 한도는 오늘 성공한 `conversion` 거래의 `hbtReceived` 합산으로 체크
- HBT 수량은 온체인 `currentRate / 1e8` 비율로 계산

근거:

- 서버 상수/검증: `functions/index.js:47-48`, `functions/index.js:69-160`

## 9-2. 처리 순서

## 9-3. 프런트와 참고 코드의 불일치

실제 사용자 UI와 서버는 12000 HBT 기준으로 동작한다.

- 자산 탭 표시: `js/app.js:1699-1700`
- 서버 강제: `functions/index.js:47`, `functions/index.js:159`

그런데 참고용 설정/헬퍼에는 예전 제한이 남아 있다.

- `js/blockchain-config.js:149` → `maxConversionPerDay: 1000`
- `js/blockchain-manager.js:344-347` → `calculateHbtWithHalving()`이 1000을 상한으로 사용

현재 코드상 이 헬퍼는 실제 변환 경로에서 호출되지 않는 것으로 보인다. 따라서 “실제 제한”은 12000이지만, 보조 코드와 설정 설명은 아직 1000 기준이 남아 있는 상태다.

## 11. 구현상 주의 포인트

### 11-1. 일일 기본 포인트는 누적형이다

- 같은 날짜 기록을 수정해도 이미 받은 기본 포인트는 회수되지 않는다.
- 운영 정책과 사용자 안내 문구가 이 구현과 맞는지 확인이 필요하다.

### 11-2. “오늘 +P”는 기본 포인트만이 아니다

- 리액션 수신/지급도 포함된다.
- 사용자가 생각하는 “저장해서 번 포인트”와 자산 탭 숫자가 다를 수 있다.

### 11-3. 설정 파일과 실제 지급 로직이 완전히 같지 않다

- 일일 HBT 한도: 참고 설정 1000, 실제 12000
- 30일 마스터 HBT 보너스: 참고 설정 100, 실제 200%

즉 분석이나 운영 문서를 만들 때는 `js/blockchain-config.js`만 보면 안 되고, 반드시 `functions/index.js`와 실제 UI 코드까지 함께 봐야 한다.

## 12. 결론

해빛스쿨의 보상 시스템은 “하루 기록 기반 기본 포인트” 위에 “온보딩/추천/마일스톤/리액션/월간 MVP/개인 챌린지/소셜 챌린지/HBT 변환”이 얹힌 다층 구조다.

핵심적으로 기억할 점은 세 가지다.

1. 기본 포인트 계산은 프런트, 실제 지급은 서버다.
2. 챌린지와 소셜 기능은 단순 추가 보상이 아니라 잠금/반환/소각까지 포함한다.
3. 일부 설정 파일은 현재 실제 지급 로직과 어긋나므로, 운영 기준은 서버 코드와 실제 UI를 우선해야 한다.
