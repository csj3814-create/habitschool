# 초대 링크로 추천 가입 + 친구 연결 처리 가이드

대상 프로젝트:
- `C:\SJ\antigravity\habitschool`

연동 챗봇 프로젝트:
- `C:\SJ\antigravity\habitchatbot`

작성일:
- 2026-04-05

---

## 목표

기존 초대 링크 `?ref=추천코드`를 성장과 소셜 둘 다에 쓰도록 정리합니다.

사용자 기대 흐름은 아래와 같습니다.

1. 초대자가 앱이나 챗봇에서 자기 초대 링크를 공유
2. 상대가 링크를 열기
3. 아직 미가입이면 추천 가입이 기록됨
4. 가입이 끝나면 초대자와 친구도 연결됨
5. 이미 가입한 회원이면 추천 보상은 건너뛰고 친구 연결만 진행됨

즉 링크 하나로 아래 두 가지를 같이 처리합니다.

- 신규 회원: `추천 가입 + 친구 연결`
- 기존 회원: `친구 연결`

---

## 제품 규칙

### 1. 신규 회원

- 링크에 `?ref=ABC123` 이 있으면 기존처럼 추천 가입 코드를 처리합니다.
- 단, 가입 완료 뒤 추천인과 새 회원을 자동으로 `active friendship` 으로 연결합니다.
- 이때 추천 보상은 기존 규칙을 그대로 유지합니다.

### 2. 기존 회원

- 이미 로그인된 회원이 링크를 열면 추천 보상은 주지 않습니다.
- 대신 앱에서 `OOO님과 친구 맺기` 확인 UI를 보여주고, 한 번 누르면 바로 친구 연결합니다.
- 기존 회원은 링크 미리보기/오동작 방지를 위해 "열자마자 자동 연결" 대신 "확인 1탭"을 권장합니다.

### 3. 자기 링크

- 자기 자신의 `referralCode` 링크를 열면 친구 연결도 추천도 처리하지 않습니다.

### 4. 이미 친구인 경우

- 이미 `active friendship` 이 있으면 no-op 처리합니다.
- 안내는 `이미 친구예요` 정도로 간단히 보여주면 충분합니다.

### 5. 기존 pending 요청이 있는 경우

- 초대 링크로 들어온 연결은 상호 의사가 충분하다고 보고 `pending` 을 `active` 로 바로 올립니다.
- 즉 `friendships/{pairId}` 가 `pending` 이어도 invite-link 경로에서는 바로 확정해도 됩니다.

## 데이터 모델 원칙

기존에 정한 구조를 그대로 유지합니다.

- 진실 원본: `friendships/{pairId}`
- 캐시: `users/{uid}.friends`

권장 `source` 값:

- `invite_link_signup`
- `invite_link_existing`
- 기존 `chatbot`, `app` 값은 유지

## 권장 구현 구조

### A. 앱 클라이언트

클라이언트는 `ref` 파라미터를 읽고 아래 두 갈래로 나눕니다.

#### 로그인 전

- `ref` 값을 localStorage 또는 메모리 상태에 저장
- 로그인/가입 완료 뒤 후처리 실행

#### 로그인 후

- 현재 로그인한 사용자가 기존 회원이면:
  - 추천 보상은 건너뜀
  - 친구 연결 확인 UI 오픈
  - 확인 시 새 callable 호출

### B. Cloud Functions

서버는 아래 2개를 담당합니다.

1. 신규 회원 추천 가입 처리
2. 기존 회원 친구 연결 처리

추천 방식:

- `processReferralSignup` 을 확장해서 친구 연결도 함께 처리
- 새 callable `acceptInviteLinkFriendship` 를 추가해서 기존 회원 흐름 처리

## 정확한 수정 포인트

### 1. `js/app.js`

아래 영역을 중심으로 수정합니다.

- 초대 링크 표시/복사/공유 영역
  - 현재 `referralUrl = ${APP_ORIGIN}?ref=${userData.referralCode}` 를 쓰는 부분 유지
- 앱 시작 시 URL 파라미터 읽는 위치
  - 로그인 상태 확인 직후
  - 또는 앱 부트스트랩 직후
- 로그인/가입 완료 직후 후처리 위치
  - 이미 `processReferralSignup` 을 호출하는 지점이 있으면 거기에 연결

권장 함수 추가:

```js
function getInviteRefFromUrl() {}
function persistPendingInviteRef(code) {}
function readPendingInviteRef() {}
function clearPendingInviteRef() {}
async function maybeHandleInviteLinkAfterAuth() {}
async function maybePromptExistingMemberInviteFriendship(code) {}
```

#### 해야 할 일

- `?ref=` 가 있으면 값을 저장
- 로그인 후 아래 분기
  - `referredBy` 가 없고 신규 가입 흐름이면 `processReferralSignup(code)` 호출
  - 이미 가입된 회원이면 `acceptInviteLinkFriendship({ referralCode: code })` 호출 전 확인 UI 노출
- 처리 후 URL 에서 `ref` 제거

### 2. `functions/index.js`

#### A. `processReferralSignup` 확장

현재 역할:

- `referralCode` 조회
- `referredBy` 저장
- 가입 보너스 지급

추가할 역할:

- 추천인과 신규 가입자를 바로 친구 연결

권장 동작:

1. `referralCode` 로 추천인 uid 조회
2. 자기 자신 추천 방지
3. 이미 `referredBy` 있으면 중복 차단
4. `referredBy` 저장 + 보너스 처리
5. `friendships/{pairId}` 를 `active` 로 upsert
6. 양쪽 `users/{uid}.friends` 에 서로 추가
7. `source: 'invite_link_signup'`

#### B. 새 callable `acceptInviteLinkFriendship`

용도:

- 기존 가입 회원이 초대 링크를 열었을 때 친구만 연결

입력 예시:

```json
{
  "referralCode": "ABC123"
}
```

권장 반환 예시:

```json
{
  "success": true,
  "status": "connected",
  "inviterUid": "uid_123",
  "inviterName": "홍길동"
}
```

처리 규칙:

1. auth 필수
2. `referralCode` 로 초대자 uid 조회
3. 자기 자신이면 차단
4. 이미 `active friendship` 이면 `already_active`
5. `pending` 이면 바로 `active` 로 승격
6. 없으면 새 `active friendship` 생성
7. 양쪽 `friends` 배열 동기화
8. `source: 'invite_link_existing'`

---

### 3. 공통 서버 헬퍼 권장

Functions 안에 아래 헬퍼를 두면 중복이 많이 줄어듭니다.

```js
function buildFriendshipId(uidA, uidB) {}
async function upsertActiveFriendship({ db, uidA, uidB, nameA, nameB, source }) {}
```

`upsertActiveFriendship()` 가 하는 일:

- `friendships/{pairId}` 생성/갱신
- `status: 'active'`
- `users` 정렬 저장
- `userNames` 저장
- `acceptedAt`, `updatedAt`, `source` 저장
- 양쪽 `users/{uid}.friends` 에 서로 `arrayUnion`

이 헬퍼를 아래 두 군데에서 같이 쓰면 됩니다.

- `processReferralSignup`
- `acceptInviteLinkFriendship`

## URL 처리 규칙

친구/추천 처리 후에는 URL 에서 `ref` 를 제거하는 걸 권장합니다.

예시:

```js
function clearInviteRefFromUrl() {
    const url = new URL(window.location.href);
    url.searchParams.delete('ref');
    window.history.replaceState({}, '', url.toString());
}
```

이유:

- 새로고침 시 중복 처리 방지
- 이미 처리된 링크가 다시 실행되지 않음

## E2E 시나리오

### 시나리오 1. 신규 회원

1. A가 앱/챗봇에서 자기 초대 링크 복사
2. B가 링크 열기
3. B가 회원가입 완료
4. `referredBy` 저장 확인
5. A-B `friendships/{pairId}.status === 'active'` 확인
6. 양쪽 `users.friends` 반영 확인

### 시나리오 2. 기존 회원

1. A가 링크 공유
2. B가 로그인된 상태로 링크 열기
3. `친구 연결하기` 확인 UI 노출
4. 확인 버튼 탭
5. `friendships/{pairId}.status === 'active'` 확인
6. 추천 보상 미지급 확인

### 시나리오 3. 자기 링크

1. 내가 내 링크 열기
2. no-op 와 안내 문구 확인

### 시나리오 4. 이미 친구

1. 이미 친구 상태에서 링크 다시 열기
2. 중복 문서 생성 없이 `already_active` 또는 no-op 확인

## 챗봇과의 계약

챗봇 쪽은 `!내코드` 에서 아래를 보여주도록 바뀝니다.

- 전체 초대 링크
- fallback 친구 코드

즉 앱 쪽에서 별도 새 링크 포맷을 만들 필요는 없습니다.
기존 `?ref=${referralCode}` 포맷을 계속 쓰면 됩니다.

## 한 줄 요약

`?ref=추천코드` 링크는 앞으로 `신규 회원에게는 추천 가입 + 친구 연결`, `기존 회원에게는 친구 연결`을 담당하고, 친구 연결의 원본은 계속 `friendships/{pairId}` 와 `users.friends` 캐시 구조를 사용합니다.
