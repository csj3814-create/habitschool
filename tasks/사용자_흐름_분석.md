- 앱 초기 진입: `js/main.js:10-41`
- Firebase/PWA 설정: `firebase.json`, `manifest.json`

## 2. 전체 사용자 여정 요약

사용자 흐름은 큰 틀에서 아래 순서로 움직인다.

1. 방문
2. 로그인 또는 게스트로 갤러리 둘러보기
3. 로그인 후 초기 데이터/온보딩/지갑/대시보드 로드
4. 프로필 설정과 건강 데이터 입력
5. 식단, 운동, 수면/마음 기록 저장
6. 저장 결과로 포인트, 스트릭, 추천 보상, 친구 알림 반영
7. 대시보드와 자산 탭에서 누적 상태 확인
8. 포인트를 HBT로 변환하고 개인 챌린지 진행
9. 갤러리에서 반응, 댓글, 친구 추가, 신고/차단 수행
10. 친구 대상 소셜 챌린지 생성 또는 초대 수락

## 3. 진입과 로그인 흐름

### 3-1. 첫 방문

- 기본 진입 화면은 로그인 모달이다.
- 단, 로그인하지 않아도 갤러리 탭은 게스트 모드로 열 수 있다.
- `?ref=XXXXXX` 형식의 추천 코드가 붙어 들어오면 즉시 `localStorage.pendingReferralCode`에 저장한다.

핵심 코드:

- 추천 코드 저장: `js/auth.js:10-13`
- 로그인 모달/게스트 버튼 UI: `index.html:110-169`

### 3-2. 구글 로그인

- 로그인은 Firebase `signInWithPopup` 기반이다.
- 인앱 브라우저(WebView)에서는 로그인 버튼을 숨기고 외부 브라우저로 열기 안내를 보여준다.
- 로그인 성공 시 페이지를 새로고침해 인증 상태를 다시 초기화한다.

핵심 코드:

- 구글 로그인 시작: `js/auth.js:100-127`

### 3-3. 로그인 후 초기화

인증 상태가 살아나면 한 번에 여러 작업이 백그라운드로 시작된다.

- 로그인 모달 숨김
- 포인트 배지와 날짜 UI 표시
- 기본 탭을 `dashboard`로 열거나 `?tab=` 값이 있으면 해당 탭 열기
- 오늘 날짜의 기록 불러오기
- 갤러리와 자산 탭 데이터 미리 불러오기

- 초대 링크/차단 목록/닉네임/건강 프로필 복원
- 지연 실행으로 온보딩, 대사건강 점수, 인바디/혈액검사 히스토리, FCM 등록 수행
- 가장 나중에 블록체인 모듈 로드, 사용자 지갑 초기화, 만료 챌린지 정산 확인

## 5. 온보딩 흐름

- 로그인 후 약 5초 뒤 `checkOnboarding()`이 호출된다.

## 6. 탭 구조와 화면 전환 흐름

앱은 탭 기반 구조다.

- `dashboard`: 오늘 상태, 스트릭, 주간 미션, 커뮤니티 요약
- `profile`: 닉네임, 인바디/건강 프로필, 혈액검사, 추천 링크, 30일 리포트
- `assets`: 포인트, HBT, 개인 챌린지, 포인트→HBT 변환
- `diet`: 식단 기록
- `exercise`: 운동/걸음수 기록
- `sleep`: 수면/명상/감사일기 기록
- `gallery`: 커뮤니티 피드

탭 전환 시 주의점:

- 비로그인 사용자는 `gallery` 외 탭 진입 시 로그인 모달이 뜬다.
- `assets` 탭은 Firestore 기반 자산을 먼저 보여주고, 블록체인 잔액은 나중에 보강한다.
- `gallery` 탭은 들어갈 때마다 `loadGalleryData()`를 호출한다.
- 기록 입력 탭은 탭 이동만으로 다시 로드하지 않고, 날짜 변경 시에만 해당 날짜 데이터를 읽는다.

핵심 코드:

- 탭 전환 메인 함수: `js/app.js:2028-2131`

## 8. 일일 기록 입력 흐름

### 8-1. 날짜 선택과 기존 기록 복원

- 사용자는 선택 날짜 기준으로 입력 화면을 본다.

- 기존 기록이 있으면 식단 사진, 운동 블록, 수면 이미지, 감사일기, 기존 AI 분석, 걸음수 데이터, 퀘스트 포인트 표시까지 복원한다.

### 8-5. 저장 버튼 동작

- `saveDataBtn`은 갤러리 탭에서는 저장이 아니라 단톡방 참여 버튼으로 동작한다.
- 기록 탭에서는 실제 저장 버튼으로 동작한다.
- 저장 시 아래 순서가 실행된다.

1. 미래 날짜 저장 차단
2. 기존 `daily_logs` 문서를 캐시/네트워크에서 읽어 이전 포인트와 URL 확보
3. 식단, 운동, 수면 업로드를 병렬 처리
4. 새 포인트 계산
5. `daily_logs/{uid}_{date}` 문서에 `merge: true`로 저장
6. 버튼 복원, 포인트 배지 즉시 갱신, 퀘스트 UI 업데이트
7. 갤러리 캐시 초기화, 마일스톤/챌린지 진행도 후처리 실행

핵심 코드:

- 저장 버튼 이벤트: `js/app.js:4123-4458`

- 주간 식단/운동/마음 달성 수
- 커뮤니티 현황
- 친구 오늘 현황
- 소셜 챌린지 카드

## 11. 자산 탭 흐름

자산 탭은 포인트와 HBT를 보여주는 개인 경제 화면이다.

### 11-1. 자산 조회

- 먼저 Firestore의 `users.coins`를 보여준다.
- HBT는 온체인 잔액이 진실의 원천이라 Firestore 값을 믿지 않고 별도 조회한다.
- 오늘 획득한 포인트와 오늘 획득/변환한 HBT 증감도 따로 계산한다.
- 최근 7일 HBT 변환 히스토리 미니차트와 오늘 변환 한도도 표시한다.

## 14. 소셜 챌린지 흐름

소셜 챌린지는 HBT 개인 챌린지와 별개의 “친구 간 포인트 경쟁/협동” 기능이다.

### 14-1. 생성 조건

- 로그인 필요
- `users.friends`에 친구가 있어야 함
- 친구는 최대 3명 선택
- 경쟁 모드는 1명만 선택
- 경쟁 모드는 포인트 스테이크 필요
- 생성자와 참가자는 최근 30일 내 최소 활동 일수 조건을 만족해야 함
- 서로 친구 등록 상태여야 함

프런트 흐름:

- 모달 열기
- 친구 목록 로드
- 유형(단체 목표 / 1:1 경쟁), 기간, 스테이크 설정
- `createSocialChallenge` 호출

핵심 코드:

### 14-2. 초대 수락/거절

- 초대받은 사용자는 초대 모달에서 수락 또는 거절한다.

- 거절이 모두 발생하면 취소되며 경쟁 모드의 생성자 스테이크는 반환된다.

## 16. 현재 구조에서 눈에 띄는 해석 포인트

### 16-1. 이 앱의 진짜 홈은 대시보드다

사용자는 기록 입력도 중요하지만, 실제 체감 흐름은 “로그인 → 대시보드 → 오늘 해야 할 것 확인 → 기록 입력”에 가깝다.

### 16-2. 저장은 프런트에서, 보상 확정은 서버에서

프런트는 `awardedPoints`를 계산해 문서를 저장하지만, 최종 코인 증가와 추천 보상은 모두 서버 트리거가 담당한다. 이 구조를 이해해야 포인트 관련 버그를 정확히 추적할 수 있다.

### 16-3. 갤러리는 단순 피드가 아니라 핵심 재참여 장치다

비로그인 사용자도 갤러리를 열 수 있고, 로그인하면 반응/댓글/친구/소셜 챌린지로 연결된다. 즉 커뮤니티 진입점 역할이 강하다.

### 16-4. 블록체인은 늦게 로드되는 보상 확장 계층이다

앱의 핵심 기록 경험은 Firebase만으로 먼저 돌아가고, HBT와 챌린지는 뒤늦게 로드되는 구조다. 그래서 사용자 체감상 “건강 기록 앱 + 토큰 기능 추가”에 가깝다.

### 16-5. 현재 온체인 연동은 BSC 테스트넷 기준으로 보인다

문서상 토큰 기능이 크지만 실제 상수는 BSC 테스트넷 탐색기와 RPC를 가리킨다. 운영/실서비스 해석 시 혼동 주의가 필요하다.

근거:

- 프런트 탐색기 링크: `js/main.js:141`
- 서버 RPC/Explorer: `functions/index.js:40-44`
